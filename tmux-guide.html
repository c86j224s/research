tmux 구조와 동작 원리

tmux가 어떻게 동작하는지, 세션·윈도우·패인의 계층 구조를 이해합니다.

🖥️ tmux server

백그라운드 데몬으로 실행.
모든 세션·윈도우·패인을 소유하고 상태를 유지.

⟷ Unix socket
(tmp/tmux-*/default)

🖱️ tmux client(s)

터미널에서 소켓으로 서버에 연결.
여러 클라이언트가 동시에 같은 세션에 Attach 가능.

핵심 원리: 터미널을 닫아도 서버 프로세스는 살아있으므로, Ctrl+b d로 분리(Detach)하면 세션 상태가 그대로 보존됩니다. SSH 연결이 끊겨도 작업이 계속 실행됩니다.
계층 구조 (Hierarchy)
Server
tmux 서버 프로세스 — 하나만 실행됨 (PID 유지)
│
Session
독립적인 작업 공간 (예: project-A, project-B) — 여러 개 가능
│
Window
탭과 유사 — 브라우저의 탭처럼 전환 가능, 번호로 식별
│
Pane
화면 분할 단위 — 하나의 윈도우 안에서 수평/수직 분할

🔌

Attach / Detach

클라이언트가 세션에서 분리(Detach)되어도 서버는 계속 실행. 나중에 다시 Attach하면 이전 상태 그대로 복원.

📡

공유 세션

두 클라이언트가 동시에 같은 세션에 Attach 가능. 페어 프로그래밍이나 라이브 데모에 활용.

🖥️

가상 터미널

각 패인은 독립된 pseudo-terminal(pty). 서로 다른 셸이나 프로그램을 동시 실행.

💾

상태 보존

SSH 연결 끊김, 터미널 종료에도 작업 지속. 장시간 실행 작업에 필수적.

실전 활용 패턴

tmux를 효과적으로 활용하는 워크플로우와 실전 팁들을 소개합니다.

원격 작업 필수 패턴

SSH 연결 끊김 방지 — Detach & Reattach

장시간 실행되는 작업(빌드, 학습, 배포 등)을 SSH 세션 안에서 그냥 실행하면 연결 끊김 시 작업이 중단됩니다. tmux 세션 안에서 실행 후 Detach하면 SSH가 끊겨도 작업이 계속됩니다.

# 서버 접속 후 세션 생성 tmux new -s training # 작업 실행 python train.py --epochs 100 # Ctrl+b d 로 Detach (SSH 끊어도 OK) # ... 나중에 SSH 재접속 후 ... tmux attach -t training # 작업 상태 확인
개발 환경 자동 구성

프로젝트 시작 스크립트

프로젝트별 tmux 레이아웃을 스크립트로 자동화하면 매번 수동으로 패인을 나누지 않아도 됩니다.

#!/bin/bash # dev.sh SESSION="myapp" # 세션이 없을 때만 생성 tmux has-session -t $SESSION 2>/dev/null if [ $? != 0 ]; then tmux new-session -d -s $SESSION -c ~/myapp # 윈도우 0: 에디터 tmux rename-window -t $SESSION:0 editor tmux send-keys -t $SESSION:0 'vim .' Enter # 윈도우 1: 서버 + 로그 (수직 분할) tmux new-window -t $SESSION:1 -n server tmux send-keys -t $SESSION:1 'npm run dev' Enter tmux split-window -h -t $SESSION:1 tmux send-keys -t $SESSION:1.1 'tail -f logs/app.log' Enter # 윈도우 2: 쉘 tmux new-window -t $SESSION:2 -n shell fi tmux attach -t $SESSION
다중 서버 동시 작업

synchronize-panes로 동시 입력

여러 서버에 동일한 명령을 동시에 실행할 때 활용합니다. 모든 패인이 같은 입력을 받습니다.

# 각 패인에서 서버 SSH 접속 # pane 0: ssh server1 # pane 1: ssh server2 # pane 2: ssh server3 # 모든 패인 동시 입력 활성화 Ctrl+b : set-window-option synchronize-panes on # 이제 입력하는 모든 내용이 3개 서버에 동시 전달 sudo systemctl restart nginx # 동기화 해제 Ctrl+b : set-window-option synchronize-panes off
페어 프로그래밍

같은 세션 공유로 실시간 협업

두 사람이 같은 tmux 세션에 Attach하면 화면을 공유하며 실시간 페어 프로그래밍이 가능합니다.

# 호스트: 세션 생성 tmux new -s pair # 게스트: 같은 세션에 Attach (읽기 전용도 가능) tmux attach -t pair # 동일한 뷰 tmux attach -t pair -r # 읽기 전용 # 별도 윈도우 크기로 독립 접속 (그룹 세션) tmux new-session -t pair # 같은 세션, 독립 뷰포트
생산성 팁

tmux + fzf 세션 퀵 스위처

fzf와 조합하면 세션/윈도우 간 퀵 스위칭이 가능합니다. .bashrc/.zshrc에 추가하세요.

# ~/.zshrc에 추가 tm() { local session session=$(tmux list-sessions -F "#{session_name}" 2>/dev/null \ | fzf --height 40% --reverse \ --preview 'tmux list-windows -t {}') if [ -n "$session" ]; then tmux attach -t "$session" || tmux new -s "$session" fi } # 사용: tm (엔터 후 세션 선택)

~/.tmux.conf 설정 가이드

tmux 시작 시 자동으로 로드되는 설정 파일입니다.

⚠ 가장 흔한 함정

설정을 붙여넣었는데 안 보인다면 — 리로드를 안 한 것

tmux 서버는 한 번 떠있으면 ~/.tmux.conf를 다시 읽지 않습니다. 파일을 수정해도 현재 attached 세션은 옛날 설정 그대로 돌고 있습니다. 색상 테마, status-bar 포맷, 키바인딩 변경 등이 안 보이는 이유의 90%는 이것입니다.

적용 방법 (위에서부터 권장):

  1. 아래 "단축키 커스터마이징" 블록의 bind r source-file ...을 먼저 conf에 넣고 한 번 리로드 → 이후엔 Ctrl+b r 한 방.
  2. CLI에서 tmux source-file ~/.tmux.conf — 현재 세션에 즉시 적용.
  3. 완전 리셋: tmux kill-server 후 tmux로 새 세션 — 모든 캐시 상태 초기화.

📌 예외: default-terminal처럼 세션 생성 시점에만 적용되는 옵션은 source-file로는 부족하고 새 세션을 만들어야 반영됩니다. 색상이 이상하면 kill-server 후 새로 시작해보세요.

📌 source-file 시 에러가 나면 그 줄부터 아래는 무시되니, 부분만 적용된 것처럼 보일 수 있습니다. tmux source ~/.tmux.conf 출력에 에러 메시지가 있는지 확인하세요.

기본 필수 설정

처음 시작할 때 반드시 설정하면 좋은 옵션들

# Prefix 키 변경: Ctrl+b → Ctrl+a (더 편한 위치) unbind C-b set -g prefix C-a bind C-a send-prefix # 256색 + True Color 지원 set -g default-terminal "tmux-256color" set -ag terminal-overrides ",xterm-256color:RGB" # 마우스 지원 (스크롤, 패인 선택, 크기 조절) set -g mouse on # 인덱스를 0이 아닌 1부터 시작 (키보드 배열상 더 편리) set -g base-index 1 setw -g pane-base-index 1 # 윈도우 닫힐 때 번호 자동 재정렬 set -g renumber-windows on # 스크롤백 버퍼 크기 (기본 2000 → 50000) set -g history-limit 50000 # Escape 키 딜레이 제거 (Vim 사용 시 필수) set -s escape-time 0 # 시스템 클립보드 통합 (OSC52 — iTerm2 / WezTerm / kitty 등 지원) set -g set-clipboard on

Extended Keys — 고급 modifier 조합 통과 (tmux 3.2+)

Ctrl+/, Ctrl+;, Ctrl+Shift+P, Shift+Enter 같이 전통 ASCII 인코딩으로 표현할 수 없는 modifier 조합을 tmux 안의 vim/nvim까지 손실 없이 전달합니다.

# 외부 터미널이 보내는 extended 키 시퀀스를 수신 (DA 응답으로 자동 협상) set -g extended-keys on # on=협상 시 수신, always=무조건 emit # 안쪽 프로그램에 emit할 때 사용할 인코딩 set -g extended-keys-format csi-u # 또는 xterm (modifyOtherKeys 형식)

왜 필요한가: 터미널은 ASCII control 코드 기반이라 Ctrl+A는 0x01로 보내지지만 Ctrl+Shift+A나 Ctrl+/는 표준 인코딩이 없어 손실됩니다. 두 옵션이 이걸 해결합니다.

두 가지 포맷:

  • csi-u — Paul Evans "fixterms" 표준. kitty / foot / WezTerm / iTerm2 3.5+ / Ghostty 등이 지원. 권장.
  • xterm — xterm modifyOtherKeys level 2. 전통 xterm 호환.

실제 효과 (tmux 안 nvim 기준):

  • Ctrl+; Ctrl+, Ctrl+. — IDE 스타일 단축키
  • Ctrl+/ — 주석 토글 / 검색
  • Ctrl+Shift+P — 커맨드 팔레트
  • Shift+Enter Ctrl+Enter — 멀티라인 입력
  • Ctrl+Tab Ctrl+Shift+Tab — 탭 전환

⚠ 주의: 터미널이 지원해야 의미가 있습니다. Terminal.app은 미지원이라 켜도 효과 없음(피해도 없음). 안쪽 프로그램도 알아야 — nvim 0.10+ 네이티브 지원, 구버전 vim은 추가 매핑 필요. 평범한 셸 작업만 한다면 체감 효과 거의 없음, 에디터·IDE 흉내 키바인딩 쓸 때 의미가 큼.

macOS에서 tmux 사용 시 알아둘 것

Terminal.app / iTerm2 환경에서 기본 단축키 일부가 동작하지 않는 케이스와 해결법, 그리고 시스템 클립보드 연동을 정리합니다.

핵심: macOS의 터미널 에뮬레이터는 기본적으로 Option(⌥)을 "특수문자 입력 키"로, Ctrl+←/→를 "Mission Control의 Spaces 이동"으로 가로챕니다. 그래서 tmux의 Ctrl+b Alt+↑↓←→ 같은 단축키가 tmux까지 도달하지 못합니다.
가장 흔한 함정

Alt(⌥) 키가 tmux에서 동작하지 않음 — 패인 리사이즈·Vim 매핑 등 Alt 전반

증상: Ctrl+b 후 Alt+↑↓←→를 눌러도 패인 크기가 바뀌지 않거나, Vim/Neovim 안에서 Alt 기반 단축키가 먹히지 않음. 원인은 macOS 터미널이 ⌥을 "Meta 키"가 아닌 "특수문자 modifier"로 해석하기 때문 — Alt 키 시퀀스 자체가 tmux에 도달하지 않습니다. 해결법은 세 가지입니다.

방법 1 — iTerm2: Option을 Esc+ (Meta)로 설정 ✓ 권장

Alt 관련 모든 문제의 근본 해결책. 이것만 해도 대부분의 Alt 키 이슈가 사라집니다.

# Settings → Profiles → Keys → General # Left Option Key: Esc+ # Right Option Key: Esc+ (선택)

방법 2 — Terminal.app: "Use Option as Meta key" 체크

# Terminal → Settings → Profiles → 사용 중인 프로파일 선택 # → Keyboard 탭 → "Use Option as Meta key" 체크

방법 3 — 권장: ⌥ 없는 tmux 키바인딩으로 교체

1·2번 설정을 켜면 ⌥+숫자/알파벳으로 입력하던 특수문자(€, ©, ñ, 한글 일부 IME 등)를 잃습니다. 다국어 입력이 잦다면 차라리 tmux 단축키를 ⌥ 없는 형태로 바꾸는 게 깔끔합니다.

# ~/.tmux.conf — Vim 스타일 반복 가능한 리사이즈 (-r 플래그) bind -r H resize-pane -L 5 bind -r J resize-pane -D 5 bind -r K resize-pane -U 5 bind -r L resize-pane -R 5 # 또는 Shift+Arrow 매핑 (직관적) bind -r S-Left resize-pane -L 5 bind -r S-Down resize-pane -D 5 bind -r S-Up resize-pane -U 5 bind -r S-Right resize-pane -R 5

Ctrl+b를 누른 직후 H/J/K/L을 반복 입력하면 prefix를 다시 누르지 않고도 계속 크기를 조절할 수 있습니다(-r = repeatable).

클립보드 통합

tmux ↔ macOS 시스템 클립보드 (pbcopy / pbpaste)

복사 모드에서 선택한 내용을 ⌘V로 다른 앱에 붙여넣고, ⌘C로 복사한 외부 텍스트를 tmux 안에서 붙여넣을 수 있게 합니다.

# ~/.tmux.conf setw -g mode-keys vi set -g set-clipboard on # OSC52 (iTerm2 / WezTerm 지원) # 키보드 복사: prefix [ → v 로 선택 시작 → y 로 복사 bind -T copy-mode-vi v send-keys -X begin-selection bind -T copy-mode-vi y send-keys -X copy-pipe-and-cancel "pbcopy" # 마우스로 드래그 → 손 떼는 순간 pbcopy bind -T copy-mode-vi MouseDragEnd1Pane \ send-keys -X copy-pipe-and-cancel "pbcopy" # prefix + p 로 pbpaste 내용 붙여넣기 bind p run "pbpaste | tmux load-buffer - && tmux paste-buffer"

iTerm2 사용자는 Settings → General → Selection에서 "Applications in terminal may access clipboard"를 켜야 set-clipboard on(OSC52)이 동작합니다. Terminal.app은 OSC52를 지원하지 않으므로 pbcopy 파이프 방식이 사실상 필수입니다.

readline 키 통과

⌘+←/→ (줄 처음/끝 이동)이 tmux 안에서 안 먹을 때

원인: ⌘(Cmd) 키는 macOS가 OS/앱용으로 가로채므로 tmux나 셸에 직접 도달하지 않습니다. ⌘+Arrow의 "줄 처음/끝" 동작은 사실 터미널 앱이 ⌘+Left를 Ctrl+A 같은 시퀀스로 변환해서 셸 readline에 전달하는 매핑입니다. tmux 안에서만 안 된다면 tmux 잘못이 아니라 터미널의 키 매핑이 빠져있다는 뜻입니다 (Ctrl+A/E는 tmux를 그대로 통과합니다).

iTerm2 — Natural Text Editing 프리셋 (가장 쉬움)

# Settings → Profiles → 프로파일 → Keys → Key Mappings # 하단의 "Presets..." → "Natural Text Editing" 선택 # 자동 매핑되는 항목: # ⌘+← → 0x01 (Ctrl+A) : 줄 처음 # ⌘+→ → 0x05 (Ctrl+E) : 줄 끝 # ⌘+Delete → 0x15 (Ctrl+U) : 줄 처음까지 삭제 # ⌥+←/→ → Esc B / Esc F : 단어 단위 이동 # ⌥+Delete → Esc Del : 단어 단위 삭제

Terminal.app — 수동 매핑

프리셋이 없어 손으로 추가합니다. Terminal → Settings → Profiles → Keyboard → Key Mappings에서 + 로:

# Modifier Key Action Value ⌘ ← Send Text \001 # Ctrl+A: 줄 처음 ⌘ → Send Text \005 # Ctrl+E: 줄 끝 ⌘ Delete Send Text \025 # Ctrl+U: 줄 처음까지 삭제 ⌥ ← Send Text \033b # Esc B: 단어 앞 ⌥ → Send Text \033f # Esc F: 단어 뒤

매핑이 적용된 뒤에는 tmux 안에서도 동일하게 동작합니다. 시퀀스 자체는 모두 단순한 control 문자나 escape 조합이라 tmux가 별다른 조작 없이 그대로 통과시킵니다.

readline 치트시트

⌘+Arrow 매핑 대신 외워두면 어디서나 통하는 단축키

readline은 bash/zsh의 라인 편집 라이브러리로, tmux·SSH·도커 컨테이너 어디서든 동일하게 동작합니다. ⌘+Arrow 매핑이 환경마다 달라 신뢰하기 어렵다면 처음부터 이 키들에 손을 익히는 게 가장 범용적입니다.

Ctrl+A
줄 처음으로 이동 (Home)
Ctrl+E
줄 끝으로 이동 (End)
Ctrl+B / Ctrl+F
한 글자 뒤/앞 이동 (← / →)
Esc B / Esc F
한 단어 뒤/앞 이동 (⌥+← / ⌥+→ 대체)
Ctrl+U
커서부터 줄 처음까지 삭제 (kill backward)
Ctrl+K
커서부터 줄 끝까지 삭제 (kill forward)
Ctrl+W
커서 직전 한 단어 삭제
Esc D
커서 직후 한 단어 삭제
Ctrl+Y
방금 삭제한 텍스트 다시 붙여넣기 (yank)
Ctrl+T
커서 좌우 두 글자 교환
Ctrl+L
화면 클리어 (clear와 동일, 입력 라인 유지)
Ctrl+R
히스토리 역방향 증분 검색
Ctrl+P / Ctrl+N
히스토리 이전/다음 (↑ / ↓)
Ctrl+C
현재 입력 취소 (실행 중이면 SIGINT)
Ctrl+D
빈 라인이면 셸 종료, 아니면 커서 위치 글자 삭제
Ctrl+Z
실행 중 프로세스를 백그라운드로 일시정지 (SIGTSTP)

📌 Esc 조합은 Esc를 누르고 뗀 뒤 다음 키를 누르는 시퀀스입니다. 터미널이 ⌥을 Meta로 인식하도록 설정되어 있다면 Alt(⌥)+B처럼 동시 입력도 동일하게 동작합니다.

📌 zsh를 vi 모드(bindkey -v)로 쓰는 경우 일부 키가 다르게 바인딩될 수 있습니다. 기본은 emacs 모드(bindkey -e)이며 위 표는 emacs 모드 기준입니다.

마우스 모드와 ⌘C

마우스 모드를 켜도 ⌘C로 직접 복사하려면 — Option 드래그

set -g mouse on이 켜져 있으면 드래그 선택을 tmux가 가로채서 ⌘C가 평소처럼 동작하지 않습니다. macOS에서는 드래그하는 동안 Option(⌥)을 누르고 있으면 터미널이 tmux를 우회해 직접 텍스트를 선택하므로 그대로 ⌘C로 시스템 클립보드에 복사됩니다 (Terminal.app · iTerm2 공통).

즉, "tmux 복사 모드 → y" 와 "Option 드래그 → ⌘C" 두 가지 경로를 상황에 맞게 쓰면 됩니다.

기타 충돌

그 외 macOS에서 잘 안 먹는 키 조합

  • Ctrl+← / Ctrl+→ — Mission Control "Spaces 간 이동" 단축키와 충돌. System Settings → Keyboard → Keyboard Shortcuts → Mission Control에서 해당 항목 해제 시 tmux로 전달됩니다. (Prefix 후 Ctrl+↑↓←→로 리사이즈하려면 필요)
  • Ctrl+Space — 기본 "입력 소스 전환" 단축키. Keyboard Shortcuts → Input Sources에서 해제 권장.
  • F1~F12 — 기본은 밝기/볼륨 등 미디어 키. Fn과 함께 누르거나 Keyboard → "Use F1, F2, etc. keys as standard function keys"를 켜야 합니다.
  • Cmd (⌘) — tmux는 ⌘를 modifier로 받을 수 없습니다(터미널이 자체적으로만 소비). 단축키 설계 시 ⌘를 가정하지 말 것.
  • Ctrl+b Prefix 자체는 macOS와 충돌이 없으나, readline의 "한 글자 뒤로(backward-char)" 단축키와 같아 셸 작업 중 잠깐 혼동이 생길 수 있습니다. 거슬리면 Prefix를 Ctrl+a 또는 Ctrl+space 등으로 바꾸세요(설정 탭의 기본 설정 참고).
