UPDATE/DELETE 주의: ClickHouse의 UPDATE는 즉각 반영되지 않습니다. 내부적으로 파트를 다시 쓰는 무거운 작업입니다. 잦은 수정이 필요한 데이터는 ClickHouse에 부적합합니다.

🌲
MergeTree
모든 MergeTree 계열의 기반 — 거의 모든 경우의 기본 선택
기본값
🎯 이 엔진을 써야만 하는 상황
데이터가 계속 쌓이기만 할 때
서버 로그, 클릭 이벤트, 결제 내역처럼 한 번 기록하면 거의 수정하지 않는 데이터. 다른 엔진들은 모두 이 위에서 동작하는 변형임.
시간 범위 기준으로 집계할 때
"지난 7일간 DAU", "이번 달 매출"처럼 시간 컬럼을 ORDER BY 앞에 두면 해당 범위 Granule만 읽어 초고속 응답.
💡 반대로 쓰면 안 되는 상황: 행 단위 UPDATE가 자주 일어나거나, 트랜잭션이 필요한 경우 → PostgreSQL/MySQL이 적합
핵심 개념
INSERT마다 정렬된 Part를 새로 생성하고, 백그라운드에서 여러 Part를 하나로 Merge합니다.

ORDER BY에 지정한 컬럼이 Primary Index 역할을 하며, 8192행(Granule)마다 인덱스 포인트 하나씩 기록합니다.

PARTITION BY로 월/일 단위로 분리하면 해당 파티션 외에는 아예 읽지 않습니다.
장점
범위 쿼리 초고속
효율적 압축
파티션 pruning
모든 기능의 베이스
단점
UPDATE/DELETE 느림
단건 INSERT 비효율
트랜잭션 없음
CREATE TABLE events (
    event_time  DateTime,
    user_id     UInt64,
    action      String,
    revenue     Float64
)
ENGINE = MergeTree()
PARTITION BY toYYYYMM(event_time)  -- 월별 파티션: 다른 달은 아예 안 읽음
ORDER BY     (event_time, user_id)   -- 이 순서로 물리 정렬 + 인덱스
SETTINGS     index_granularity = 8192 -- Granule 크기 (기본값)
🔁
CollapsingMergeTree
sign 컬럼으로 이전 값을 "취소"해서 논리 삭제/수정을 구현
논리 삭제
🎯 이 엔진을 써야만 하는 상황
상태가 자주 바뀌는데 DELETE가 너무 느릴 때
예: 사용자 세션 상태(active/inactive), 주문 상태(pending→shipped→done). MergeTree에서 DELETE는 비동기라 즉각 반영이 안 됨. CollapsingMergeTree는 INSERT만으로 수정처럼 동작.
이벤트 소싱(Event Sourcing) 패턴을 쓸 때
각 상태 변화를 +1/-1 이벤트로 기록하고, 집계 시점에 현재 상태를 계산. Kafka와 조합하면 이전 상태 취소 + 새 상태 추가를 스트리밍으로 처리 가능.
💡 반대로 쓰면 안 되는 상황: 삽입 순서가 보장되지 않으면 +1/-1 쌍이 뒤집혀 엉킬 수 있음 → 이 경우 VersionedCollapsingMergeTree 사용
핵심 개념 — Sign 컬럼
각 행에 Sign 컬럼(Int8)을 두고, +1 = 데이터 추가, -1 = 데이터 취소를 의미합니다.

Merge 시 같은 키의 +1/-1 쌍을 서로 상쇄(collapse)해 둘 다 제거합니다.

즉, "이전 상태 취소(-1) + 새 상태 추가(+1)" 두 행을 INSERT해서 UPDATE처럼 씁니다.
장점
DELETE 없이 논리 삭제
상태 변경 이력 추적
고빈도 상태 변경에 적합
단점
Merge 전 중간 상태 존재
쿼리에 sign 고려 필수
삽입 순서 보장 필요
동작 시각화
-- 주문 상태: pending
order=1, status=pending, sign=+1
-- 상태 변경: shipped로
order=1, status=pending, sign=-1 ← 취소
order=1, status=shipped, sign=+1 ← 새 값
↓ Merge 후
order=1, status=shipped ✓
CREATE TABLE orders (
    order_id  UInt64,
    status    String,
    sign      Int8   -- +1 또는 -1
) ENGINE = CollapsingMergeTree(sign)
  ORDER BY order_id

-- Merge 전에도 정확한 집계:
SELECT order_id,
  argMax(status, sign)
FROM orders
GROUP BY order_id
HAVING sum(sign) > 0
VersionedCollapsingMergeTree: version 컬럼을 추가해 삽입 순서 무관하게 올바른 Collapse를 보장. 병렬/비순서 INSERT 환경에서 더 안전합니다.
➕
SummingMergeTree
Merge 시 숫자를 자동으로 더해서 행 수를 줄임 — 카운터/누적 집계 전용
사전 집계
🎯 이 엔진을 써야만 하는 상황
원본 이벤트가 너무 많아 집계가 느릴 때
하루 10억 건 클릭 이벤트에서 "국가별 일별 클릭 수"를 매번 집계하면 느림. SummingMergeTree로 (date, country) 키로 자동 누적하면 조회 테이블 행 수가 극적으로 줄어듦.
합산만 하면 되는 지표일 때
페이지뷰 수, 매출 합계, 이벤트 카운트처럼 SUM만 필요한 경우. AggregatingMergeTree보다 훨씬 단순하고 설계 비용이 낮음. avg나 unique 카운트가 필요하면 그때 AggregatingMergeTree로.
💡 반대로 쓰면 안 되는 상황: avg, min, max, 유니크 카운트가 필요하면 불가 → AggregatingMergeTree로. 원본 이벤트를 나중에 다시 볼 필요가 있으면 원본 MergeTree와 병행 운영.
핵심 개념
같은 ORDER BY 키를 가진 행들이 Merge될 때, 숫자 컬럼(UInt, Int, Float)을 모두 자동으로 더합니다.

특정 컬럼만 합산하려면 엔진 파라미터로 지정할 수 있습니다.

원본 이벤트를 그대로 쌓는 대신, 집계 결과만 보관하는 집계 테이블로 활용합니다.
장점
스토리지 대폭 절약
집계 쿼리 압도적으로 빠름
설계가 단순함
단점
avg, min, max 불가
Merge 전 중복 행 존재
원본 복원 불가
10억 행 → 수천 행으로 압축
-- 원본 이벤트 (10억 행)
2024-01-01 | KR | clicks=1
2024-01-01 | KR | clicks=1
2024-01-01 | KR | clicks=1 ...
↓ Merge 후 자동 합산
2024-01-01 | KR | clicks=340,821,042
2024-01-01 | US | clicks=128,442,001
행 수: 10억 → 국가 수 × 날짜 수
CREATE TABLE clicks_daily (
    date     Date,
    country  String,
    clicks   UInt64,
    revenue  Float64
) ENGINE = SummingMergeTree()
  ORDER BY (date, country)
-- 또는 지정 컬럼만:
-- SummingMergeTree(clicks, revenue)

-- Merge 전 중복 대비: 항상 sum()으로
SELECT date, country,
    sum(clicks)  AS total_clicks,
    sum(revenue) AS total_revenue
FROM clicks_daily
GROUP BY date, country
🔄
ReplacingMergeTree
같은 키로 다시 INSERT하면 이전 행을 덮어씀 — ClickHouse식 Upsert
Upsert
🎯 이 엔진을 써야만 하는 상황
외부 DB의 최신 상태를 ClickHouse에 싱크할 때
MySQL의 users 테이블을 ClickHouse로 복제할 때, 변경된 행을 그냥 INSERT하면 됨. Merge 후 최신 행만 남음. CDC(Change Data Capture) 파이프라인의 목적지로 최적.
행 단위 최신 상태만 필요하고 이력은 불필요할 때
사용자 프로필, 상품 정보, 설정값처럼 "지금 이 값이 뭔가"만 알면 되는 경우. CollapsingMergeTree보다 설계가 단순하고, 쿼리에 sign 처리가 없어도 됨.
💡 반대로 쓰면 안 되는 상황: 변경 이력을 보존해야 하면 불가 (이전 값이 사라짐). 동시에 많은 UPDATE가 발생하면 FINAL 조회 성능이 저하됨.
핵심 개념
같은 ORDER BY 키를 가진 행이 여러 개 있으면, Merge 시 하나만 남기고 나머지를 제거합니다.

ver 컬럼을 지정하면 가장 높은 version 행이 남고, 지정하지 않으면 가장 마지막으로 삽입된 행이 남습니다.

Merge 전까지는 중복이 존재하므로, 정확한 조회를 위해 FINAL 또는 argMax 집계가 필요합니다.
장점
Upsert를 INSERT만으로
중복 데이터 자동 정리
sign 처리 불필요
단점
Merge 전 중복 존재
FINAL은 성능 저하 큼
이력 추적 불가
CDC 싱크 시나리오
-- MySQL 변경 감지 (Debezium 등)
INSERT: user=1, name=Alice, ver=1
-- 이름 변경 → 그냥 다시 INSERT
INSERT: user=1, name=Alice2, ver=2
INSERT: user=1, name=Alice3, ver=3
↓ Merge 후 (ver 가장 높은 것 생존)
user=1, name=Alice3 ✓
CREATE TABLE users (
    user_id    UInt64,
    name       String,
    email      String,
    updated_at DateTime
) ENGINE = ReplacingMergeTree(updated_at)
  ORDER BY user_id  -- PK처럼 동작

-- 방법 ①: FINAL (정확하나 느림)
SELECT * FROM users FINAL

-- 방법 ②: argMax (빠름, 권장)
SELECT user_id,
  argMax(name,  updated_at) AS name,
  argMax(email, updated_at) AS email
FROM users GROUP BY user_id

MergeTree 내부 동작
데이터가 어떻게 쓰이고, 읽히고, 병합되는가
테이블 생성 예시
CREATE TABLE events (
    event_time  DateTime,
    user_id     UInt64,
    action      String,
    country     String,
    revenue     Float64
)
ENGINE = MergeTree()
PARTITION BY toYYYYMM(event_time)   -- 월별 파티션
ORDER BY (event_time, user_id)       -- 정렬 키 = 기본 인덱스
SETTINGS index_granularity = 8192   -- 그래뉼 크기 (기본값)

INSERT → 새 Part 생성
INSERT가 들어오면 즉시 정렬된 새 Part(디렉토리)로 디스크에 씀. 기존 데이터 수정 없음.
all_1_1_0
INSERT ①
all_2_2_0
INSERT ②
all_3_3_0
INSERT ③

백그라운드 Merge
ClickHouse가 자동으로 작은 Part들을 하나의 큰 Part로 병합. 이 과정에서 정렬 유지, 중복 제거(ReplacingMergeTree), 합산(SummingMergeTree) 등이 일어남.
all_1_1_0
+
all_2_2_0
+
all_3_3_0
→
all_1_3_1
Merged!
3
Primary Index (Sparse Index)
일반 DB와 달리 행마다 인덱스가 아닌, Granule(8192행)마다 하나씩 인덱스 포인트. 인덱스 파일이 메모리에 통째로 올라감.
-- primary.idx 내용 (개념적 표현)
Granule 0:  event_time=2024-01-01 00:00:00, user_id=1001  → 파일 오프셋 0
Granule 1:  event_time=2024-01-01 02:16:12, user_id=4821  → 파일 오프셋 X
Granule 2:  event_time=2024-01-01 05:33:07, user_id=9204  → 파일 오프셋 Y
-- 8192행에 한 번씩만 기록 → 인덱스 파일이 매우 작음

쿼리 실행 흐름
WHERE 조건이 주어지면 Primary Index로 해당 Granule 범위만 특정하고, 필요한 컬럼 파일만 읽음.
SELECT count(), sum(revenue)
FROM events
WHERE event_time BETWEEN '2024-01-01' AND '2024-01-02'
AND country = 'KR'

-- 실행 순서:
-- 1. primary.idx 에서 해당 날짜 범위 Granule 식별
-- 2. event_time.bin, country.bin, revenue.bin 만 읽음
-- 3. country='KR' 필터링 (SIMD 벡터 연산)
-- 4. revenue 합산 반환

MergeTree 변형 엔진 동작 차이
-- ReplacingMergeTree: 같은 키의 최신 행만 남김
ENGINE = ReplacingMergeTree(version)   -- version이 클수록 최신
SELECT * FROM table FINAL              -- FINAL: Merge 강제 적용

-- SummingMergeTree: 숫자 컬럼 자동 합산
ENGINE = SummingMergeTree()
-- user_id=1, revenue=100 이 두 번 들어오면
-- Merge 후 → user_id=1, revenue=200

-- AggregatingMergeTree: 집계 상태 저장
ENGINE = AggregatingMergeTree()
SELECT avgMerge(avg_state) FROM agg_table  -- -Merge 함수로 읽음
엔진 비교 & 선택 가이드
상황별 어떤 엔진을 써야 하는가
엔진 쓰기 속도 읽기 속도 UPDATE/DELETE 중복 처리 적합한 상황 주의사항
MergeTree 빠름 매우 빠름 느림 없음 로그, 이벤트, 시계열 배치 INSERT 필수
CollapsingMergeTree 빠름 보통 sign=-1로 논리 삭제 +1/-1 쌍 상쇄 상태 변경 추적, 논리 삭제 쿼리에 sign 처리 필수
ReplacingMergeTree 빠름 보통 간접 가능 키 기준 최신 최신 상태 유지 필요 FINAL은 성능 저하 큼
SummingMergeTree 빠름 매우 빠름 느림 SUM 합산 카운터, 매출 집계 avg 계산 불가
AggregatingMergeTree 보통 매우 빠름 느림 집계 상태 병합 실시간 대시보드 설계 복잡, -State 함수
Memory 극도로 빠름 극도로 빠름 빠름 없음 임시 테이블, 캐시 재시작 시 소멸
Log 빠름 느림 없음 없음 소규모, 임시 저장 인덱스 없음
Distributed 보통 빠름 (분산) 샤드 의존 없음 멀티노드 클러스터 네트워크 오버헤드
🗺 상황별 선택 플로우
데이터를 저장하고 싶다
│
├── 재시작 후에도 유지해야 하나?
│   └── NO  → Memory
│
├── 데이터가 작고 단순한가? (수백만 행 이하)
│   └── YES → Log / TinyLog
│
├── 여러 노드에 분산 저장이 필요한가?
│   └── YES → ReplicatedMergeTree + Distributed
│
└── 대용량 분석 데이터인가? (기본 케이스)
    │
    ├── 행을 업데이트(최신 상태 유지)해야 하나?
    │   └── YES → ReplacingMergeTree
    │
    ├── 논리적 삭제 / 상태 변경 이력이 필요한가?
    │   └── YES → CollapsingMergeTree (또는 VersionedCollapsingMergeTree)
    │
    ├── 숫자를 계속 누적(카운터)해야 하나?
    │   └── YES → SummingMergeTree
    │
    ├── avg, uniq 등 복잡한 사전 집계가 필요한가?
    │   └── YES → AggregatingMergeTree + Materialized View
    │
    └── 그냥 로그/이벤트 쌓고 분석
        └── MergeTree  ← 기본 선택
💡 실무 패턴: Materialized View + AggregatingMergeTree
-- 원본 테이블 (데이터 적재)
CREATE TABLE events ENGINE = MergeTree() ...

-- 집계 테이블 (사전 계산 결과 저장)
CREATE TABLE events_agg (
    date     Date,
    country  String,
    cnt      AggregateFunction(count, UInt64),
    rev_sum  AggregateFunction(sum,   Float64)
) ENGINE = AggregatingMergeTree()
  ORDER BY (date, country)

-- Materialized View (INSERT 시 자동 집계)
CREATE MATERIALIZED VIEW events_agg_mv
TO events_agg AS
SELECT
    toDate(event_time) AS date,
    country,
    countState()       AS cnt,
    sumState(revenue)  AS rev_sum
FROM events
GROUP BY date, country

-- 쿼리: 대시보드에서 초고속 조회
SELECT
    date, country,
    countMerge(cnt)    AS total_events,
    sumMerge(rev_sum)  AS total_revenue
FROM events_agg
GROUP BY date, country
원본 테이블에 INSERT 한 번으로 집계 테이블이 자동 갱신됩니다. 대시보드 쿼리는 수십억 행이 아닌 집계 결과만 읽어 수ms 응답이 가능합니다.
외부 연동 엔진
Kafka, S3, PostgreSQL 등 외부 시스템을 ClickHouse 테이블처럼 다루기
이 엔진들은 데이터를 직접 저장하지 않습니다. 외부 시스템을 SQL 테이블처럼 읽고 쓰는 어댑터 역할입니다. 보통 MergeTree 테이블과 조합해서 씁니다.

실시간 수집
동작 방식

-- 1. Kafka 엔진 테이블 (토픽 연결)
CREATE TABLE kafka_events (
    event_time  DateTime,
    user_id     UInt64,
    action      String
) ENGINE = Kafka
  SETTINGS
    kafka_broker_list    = 'kafka:9092',
    kafka_topic_list     = 'user-events',
    kafka_group_name     = 'clickhouse-consumer',
    kafka_format         = 'JSONEachRow',   -- 또는 Avro, CSV 등
    kafka_num_consumers  = 4;              -- 병렬 컨슈머 수

-- 2. 영구 저장용 MergeTree 테이블
CREATE TABLE events ENGINE = MergeTree()
ORDER BY (event_time, user_id);

-- 3. Materialized View: Kafka → MergeTree 자동 적재
CREATE MATERIALIZED VIEW kafka_to_events TO events AS
SELECT * FROM kafka_events;
-- 이 시점부터 Kafka 메시지가 자동으로 events 테이블에 쌓임

-- S3 엔진: S3의 파일을 직접 쿼리
CREATE TABLE s3_logs (
    ts      DateTime,
    level   String,
    message String
) ENGINE = S3(
    'https://s3.amazonaws.com/mybucket/logs/*.parquet',
    'ACCESS_KEY', 'SECRET_KEY',
    'Parquet'   -- CSV, JSONEachRow, Parquet, ORC 등
);

-- 그냥 SELECT하면 S3에서 직접 읽음
SELECT level, count() FROM s3_logs WHERE ts > '2024-01-01' GROUP BY level;

-- S3Queue: S3에 새 파일 추가될 때마다 자동 수집

SETTINGS mode = 'ordered'   -- 파일 순서대로 처리
S3 엔진은 데이터 레이크(Data Lake) 패턴에서 자주 씁니다. S3에 Parquet으로 쌓인 데이터를 ClickHouse에서 직접 쿼리하거나, MergeTree로 가져와 고속 분석하는 파이프라인을 구성합니다.

CREATE TABLE pg_users ENGINE = PostgreSQL(

Materialized View
INSERT 시 자동으로 집계·변환을 실행하는 ClickHouse의 핵심 패턴
일반 DB의 View는 조회할 때마다 쿼리를 실행합니다. ClickHouse의 Materialized View는 INSERT할 때 실행되어 결과를 별도 테이블에 저장합니다. 조회는 미리 계산된 결과만 읽으면 됩니다.
🧠 핵심 개념 — 언제 실행되나
일반 View
사용자가 SELECT 할 때마다
SELECT → 원본 10억 행 스캔 → 집계
SELECT → 원본 10억 행 스캔 → 집계
SELECT → 원본 10억 행 스캔 → 집계
매번 느림 ❌
Materialized View
데이터가 INSERT될 때 한 번
INSERT → 집계 → 결과 저장 ✓
조회는 결과 테이블만 읽음
SELECT → 수천 행 → 즉시 응답 ✓
SELECT → 수천 행 → 즉시 응답 ✓
계산 한 번, 조회는 항상 빠름 ✅
🏗 구조 — 항상 3개가 세트
①
원본 테이블 (Source)
실제 이벤트/로그가 INSERT되는 테이블. 보통 MergeTree.
ENGINE = MergeTree()
↓ INSERT 감지
②
Materialized View (트리거)
원본 테이블의 INSERT를 감지해 SELECT 쿼리를 실행. 결과를 대상 테이블에 씀.
CREATE MATERIALIZED VIEW ... TO ...
↓ 결과 저장
③
대상 테이블 (Target)
집계 결과가 누적되는 테이블. 보통 SummingMergeTree 또는 AggregatingMergeTree.
ENGINE = SummingMergeTree()
📦 패턴 1 — MergeTree + SummingMergeTree (단순 카운터)
원본 클릭 이벤트를 실시간으로 "국가별 일별 합산"으로 변환. 가장 단순하고 자주 쓰이는 패턴.
-- ① 원본 테이블: 이벤트 그대로 적재
CREATE TABLE clicks_raw (
    ts      DateTime,
    country String,
    revenue Float64
) ENGINE = MergeTree() ORDER BY ts

-- ② 대상 테이블: 집계 결과를 누적
CREATE TABLE clicks_daily (
    date    Date,
    country String,
    clicks  UInt64,
    revenue Float64
) ENGINE = SummingMergeTree() ORDER BY (date, country)

-- ③ MV: clicks_raw에 INSERT 시 자동 실행
CREATE MATERIALIZED VIEW clicks_mv
TO clicks_daily AS
SELECT
    toDate(ts)   AS date,
    country,
    1            AS clicks,   -- 행마다 1씩 → SummingMergeTree가 합산
    revenue
FROM clicks_raw

-- 이제 clicks_raw에 INSERT만 하면 clicks_daily가 자동 갱신됨
INSERT INTO clicks_raw VALUES (now(), 'KR', 4500)

-- 대시보드 쿼리: 초고속
SELECT date, country, sum(clicks), sum(revenue)
FROM clicks_daily GROUP BY date, country
🔬 패턴 2 — MergeTree + AggregatingMergeTree (avg, uniq 포함)
SUM만으로 안 되는 avg, 유니크 카운트 등을 사전 계산할 때. 설계가 복잡하지만 쿼리 속도는 극한으로 빠름.
-- ① 원본 테이블 (동일)
CREATE TABLE events_raw (...) ENGINE = MergeTree() ...

-- ② 대상 테이블: AggregateFunction 타입으로 상태 저장
CREATE TABLE events_agg (
    date        Date,
    country     String,
    cnt         AggregateFunction(count,  UInt64),
    uniq_users  AggregateFunction(uniq,   UInt64),
    avg_revenue AggregateFunction(avg,    Float64)
) ENGINE = AggregatingMergeTree() ORDER BY (date, country)

-- ③ MV: -State 함수로 집계 상태를 바이너리로 저장
CREATE MATERIALIZED VIEW events_agg_mv TO events_agg AS
SELECT
    toDate(ts)        AS date,
    country,
    countState()      AS cnt,
    uniqState(user_id) AS uniq_users,
    avgState(revenue)  AS avg_revenue
FROM events_raw GROUP BY date, country

-- 쿼리: -Merge 함수로 상태를 최종값으로 변환
SELECT
    date, country,
    countMerge(cnt)       AS total_events,
    uniqMerge(uniq_users)  AS unique_users,
    avgMerge(avg_revenue)  AS avg_revenue
FROM events_agg
GROUP BY date, country
-State / -Merge 함수 쌍: countState()로 저장 → countMerge()로 읽기. uniqState() → uniqMerge(), avgState() → avgMerge(). 저장 시에는 바이너리 상태로 압축되어 있다가 읽을 때 최종값으로 계산됩니다.
⚠️ Materialized View 사용 시 주의사항
알아야 할 제약
• MV 생성 이후에 INSERT된 데이터만 반영됨
  → 과거 데이터는 수동으로 채워야 함
• 원본 테이블 스키마 변경 시 MV도 재생성 필요
• INSERT 실패 시 MV도 함께 실패 (트랜잭션처럼 묶임)
• MV가 많을수록 INSERT 비용 증가
과거 데이터 채우기
-- MV 생성 전 데이터 수동 채우기
INSERT INTO clicks_daily
SELECT
    toDate(ts), country,
    count(), sum(revenue)
FROM clicks_raw
WHERE ts < '2024-01-01'  -- MV 생성 이전 기간
GROUP BY 1, 2
🎯 Materialized View를 써야만 하는 상황
실시간 대시보드
초당 수천 건 INSERT 중에도 대시보드가 ms 단위 응답. 매번 전체 집계하는 대신 MV가 미리 누적.
다차원 집계
같은 원본으로 "국가별", "디바이스별", "시간대별" 등 여러 집계 테이블을 동시에 유지. MV를 여러 개 붙이면 됨.
Kafka 수집 파이프라인
Kafka 엔진 테이블 → MV → MergeTree 조합으로 스트리밍 수집과 집계를 동시에. 별도 컨슈머 서버 없이 ClickHouse 안에서 완결.
