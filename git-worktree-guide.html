🌲 Git Worktree란?

git worktree는 하나의 Git 저장소에서 여러 개의 작업 디렉토리(working tree)를 동시에 사용할 수 있게 해주는 기능입니다. 각 worktree는 서로 다른 브랜치를 체크아웃한 독립적인 작업 공간이지만, 내부적으로는 같은 .git 저장소를 공유합니다.

일반적으로 Git 저장소에서는 한 번에 하나의 브랜치만 체크아웃할 수 있습니다. 다른 브랜치로 전환하려면 git checkout 또는 git switch를 사용해야 하는데, 이때 현재 작업 중인 변경사항을 stash하거나 commit해야 합니다. Worktree는 이 문제를 근본적으로 해결합니다.

일반 워크플로우 vs Worktree 워크플로우

❌ 기존 방식 (단일 워킹 트리)

  • 브랜치 전환 시 stash/commit 필요
  • 빌드 캐시, node_modules 등 재설치
  • IDE가 파일 변경 감지 후 재인덱싱
  • 컨텍스트 전환 비용이 큼
  • 긴급 핫픽스 시 현재 작업 중단

✅ Worktree 방식 (다중 워킹 트리)

  • 각 브랜치별 독립 디렉토리
  • 빌드 캐시, 의존성 별도 유지
  • IDE 창 여러 개로 동시 작업
  • 컨텍스트 전환 = 디렉토리 이동
  • 핫픽스를 별도 디렉토리에서 즉시 처리

Clone과의 차이점

"그냥 저장소를 여러 번 clone하면 되지 않나?" 라고 생각할 수 있습니다. 차이점을 비교해보면:

Multiple Clone

  • 디스크 공간을 N배 사용
  • 각 clone이 독립적인 remote 추적
  • 한 clone의 commit이 다른 clone에 안 보임
  • fetch를 각각 따로 해야 함
  • .git 디렉토리가 N개 존재

Worktree

  • Git objects를 공유하여 디스크 절약
  • 하나의 remote 설정 공유
  • commit 즉시 모든 worktree에서 접근 가능
  • fetch 한 번이면 전체 반영
  • .git 저장소는 하나, 참조만 공유

⚡ 핵심 명령어

Git Worktree의 주요 명령어를 알아봅니다. 각 명령어를 펼쳐서 상세 사용법과 예제를 확인하세요.

필수 git worktree add — 새 워크트리 생성

▼

새로운 워크트리를 생성합니다. 지정한 경로에 새 디렉토리를 만들고, 브랜치를 체크아웃합니다.

기본 사용법 
# 기존 브랜치로 워크트리 생성
git worktree add ../wt-feature feature-branch

# 새 브랜치를 만들면서 워크트리 생성 (-b 옵션)
git worktree add -b new-feature ../wt-new-feature

# 특정 커밋에서 새 브랜치 생성
git worktree add -b hotfix-v2 ../wt-hotfix origin/main

# detached HEAD 상태로 워크트리 생성
git worktree add --detach ../wt-temp v2.1.0

주요 옵션

옵션설명
-b <branch>새 브랜치를 생성하면서 워크트리를 만듦
-B <branch>브랜치가 이미 존재하면 리셋하고 생성
--detach브랜치 없이 detached HEAD 상태로 생성
--track원격 브랜치를 추적하도록 설정
--lock생성과 동시에 잠금 (이동식 미디어용)
--quiet출력 억제
💡 팁

경로만 지정하고 브랜치를 생략하면, 경로의 마지막 디렉토리 이름과 같은 브랜치를 체크아웃합니다.
예: git worktree add ../feature-login → feature-login 브랜치를 체크아웃

조회 git worktree list — 워크트리 목록 확인

▼

현재 저장소에 연결된 모든 워크트리의 목록을 출력합니다.

shell 
# 간단한 목록 출력
git worktree list
# 출력 예시:
# /home/user/my-project        abc1234 [main]
# /home/user/wt-feature         def5678 [feature-branch]
# /home/user/wt-hotfix          ghi9012 [hotfix-v2]

# 상세 정보 출력 (porcelain 형식)
git worktree list --porcelain

# 포맷 지정 (Git 2.36+)
git worktree list --format '%(worktree) %(HEAD) %(branch)'
ℹ️ 정보

--porcelain 형식은 스크립트에서 파싱하기 좋습니다. 각 워크트리의 경로, HEAD 커밋, 브랜치 정보를 구조화된 형태로 출력합니다.

삭제 git worktree remove — 워크트리 제거

▼

더 이상 필요 없는 워크트리를 안전하게 제거합니다. 디렉토리와 관련 메타데이터를 함께 정리합니다.

shell 
# 워크트리 제거 (클린한 상태일 때)
git worktree remove ../wt-feature

# 변경사항이 있어도 강제 제거
git worktree remove --force ../wt-feature

# 잠긴 워크트리 강제 제거 (--force 두 번)
git worktree remove --force --force ../wt-locked
⚠️ 주의

커밋되지 않은 변경사항이나 추적되지 않는 파일이 있으면 remove가 실패합니다. 중요한 작업을 잃지 않으려면 반드시 확인 후 --force를 사용하세요.

변경 git worktree move — 워크트리 이동

▼

워크트리의 경로를 변경합니다. 디렉토리를 이동하면서 메타데이터도 함께 업데이트합니다.

shell 
# 워크트리 경로 변경
git worktree move ../wt-feature ../wt-feature-v2

# 잠긴 워크트리 이동 (--force 필요)
git worktree move --force ../wt-locked ../wt-new-path
ℹ️ 참고

메인 워크트리(원본 저장소)는 이동할 수 없습니다. move는 linked worktree에만 사용 가능합니다.

정리 git worktree prune — 고아 워크트리 정리

▼

삭제된 워크트리 디렉토리의 메타데이터를 정리합니다. 디렉토리를 수동으로 rm -rf한 경우 유용합니다.

shell 
# 고아 워크트리 메타데이터 정리
git worktree prune

# 정리 전 미리보기 (dry run)
git worktree prune --dry-run

# 상세 출력과 함께 정리
git worktree prune --verbose
💡 팁

워크트리를 제거할 때는 가능하면 git worktree remove를 사용하세요. 디렉토리와 메타데이터를 한 번에 정리합니다. prune은 비정상 종료 후 복구용입니다.

보호 git worktree lock / unlock — 잠금 관리

▼

워크트리를 잠금 처리하여 prune에 의해 실수로 정리되지 않도록 보호합니다. 이동식 드라이브의 워크트리에 유용합니다.

shell 
# 워크트리 잠금
git worktree lock ../wt-feature

# 잠금 이유 메모와 함께
git worktree lock --reason "USB 드라이브에 있음" ../wt-feature

# 잠금 해제
git worktree unlock ../wt-feature

복구 git worktree repair — 연결 복구

▼

메인 저장소나 워크트리를 수동으로 이동한 후 깨진 연결을 복구합니다. (Git 2.30+)

shell 
# 메인 저장소에서 실행하여 모든 링크 복구
git worktree repair

# 특정 워크트리 경로를 지정하여 복구
git worktree repair ../wt-moved-feature

🔧 동작 원리

Git Worktree가 내부적으로 어떻게 동작하는지 이해하면, 문제 해결과 최적화에 도움이 됩니다.

.git 파일 vs .git 디렉토리

메인 워크트리에는 실제 .git 디렉토리가 존재합니다. 반면, linked worktree에는 .git 파일이 존재하며, 이 파일은 메인 저장소의 .git/worktrees/<name>을 가리킵니다.

linked worktree의 .git 파일 내용 
# cat ../wt-feature/.git
gitdir: /home/user/my-project/.git/worktrees/wt-feature
메인 저장소의 worktrees 디렉토리 구조 
.git/worktrees/wt-feature/
├── HEAD          # 이 워크트리의 현재 HEAD
├── index         # 이 워크트리의 스테이징 영역
├── gitdir        # 워크트리의 .git 파일 경로 (역참조)
├── commondir     # 공유 .git 디렉토리 경로
├── MERGE_HEAD    # (머지 중일 때)
├── REBASE_HEAD   # (리베이스 중일 때)
└── refs/         # 워크트리별 refs (bisect 등)

공유되는 것 vs 독립적인 것

🔗 모든 워크트리가 공유

  • Objects (commits, trees, blobs)
  • Refs (branches, tags)
  • Remote 설정
  • Hooks (git hooks)
  • Config (.git/config)
  • Packed-refs

🔒 각 워크트리별 독립

  • HEAD (현재 체크아웃 브랜치)
  • Index (스테이징 영역)
  • Working tree (작업 파일들)
  • MERGE_HEAD, REBASE_HEAD 등
  • Bisect 상태
  • 워크트리별 refs

Bare Repository에서의 Worktree

Bare 저장소는 워킹 트리가 없으므로, worktree를 사용하면 매우 깔끔한 구조를 만들 수 있습니다. 서버 환경이나 고급 워크플로우에서 자주 사용되는 패턴입니다.

Bare repo + Worktree 패턴 
# Bare 저장소 클론
git clone --bare https://github.com/user/repo.git repo.git
cd repo.git

# 각 브랜치별 워크트리 생성
git worktree add ../repo-main main
git worktree add ../repo-develop develop
git worktree add -b feature-x ../repo-feature-x

# 결과 디렉토리 구조:
# repo.git/          ← bare 저장소 (object store)
# repo-main/         ← main 브랜치 워크트리
# repo-develop/      ← develop 브랜치 워크트리
# repo-feature-x/    ← feature-x 브랜치 워크트리
💡 장점

Bare repo 방식은 "메인 워크트리"라는 개념이 없어서 모든 브랜치가 동등하게 취급됩니다. 깔끔하고 대칭적인 구조를 선호한다면 이 패턴을 추천합니다.

PR 코드 리뷰

동료의 PR을 별도 워크트리에서 체크아웃하여 로컬에서 실행·테스트하며 리뷰

🔀

두 브랜치의 동작을 동시에 실행하여 비교. A/B 테스트나 성능 비교에 활용

🚀

CI/CD 파이프라인

CI 서버에서 병렬 빌드 시 각 브랜치를 별도 worktree로 체크아웃하여 빌드

📋

장기 실행 태스크

테스트 스위트가 실행 중인 동안 다른 워크트리에서 개발을 계속 진행

🏷️

릴리스 관리

릴리스 브랜치를 별도 워크트리로 유지하며 메인 개발과 릴리스 준비를 병행

🔥 시나리오 1: 긴급 핫픽스

feature 브랜치에서 작업 중인데, 프로덕션에서 긴급 버그가 발견된 상황:

전체 워크플로우 
# 현재 상태: feature-login 브랜치에서 작업 중
# 프로덕션 긴급 버그 발생!

# 1. 핫픽스용 워크트리 생성
git worktree add -b hotfix/critical-bug ../hotfix origin/main

# 2. 핫픽스 워크트리로 이동
cd ../hotfix

# 3. 버그 수정, 테스트, 커밋
vim src/critical-module.js
npm test
git add -A && git commit -m "fix: 결제 실패 시 세션 복구 로직 수정"
git push origin hotfix/critical-bug

# 4. 원래 작업으로 복귀
cd ../my-project
# feature-login 브랜치가 그대로! stash 필요 없음

# 5. 핫픽스 머지 후 워크트리 정리
git worktree remove ../hotfix

👀 시나리오 2: PR 코드 리뷰

동료의 PR을 로컬에서 실행하며 리뷰하는 워크플로우:

PR 리뷰 워크플로우 
# 1. PR 브랜치를 가져오기
git fetch origin feature/new-dashboard

# 2. 리뷰용 워크트리 생성
git worktree add ../review-dashboard origin/feature/new-dashboard

# 3. 리뷰 워크트리에서 빌드 & 실행
cd ../review-dashboard
npm install && npm run dev

# → 브라우저에서 확인하며 리뷰
# → 내 작업은 원래 디렉토리에서 계속 진행 가능

# 4. 리뷰 완료 후 정리
cd ../my-project
git worktree remove ../review-dashboard

🚀 시나리오 3: CI 파이프라인에서의 활용

CI/CD 환경에서 병렬 빌드 시 worktree를 활용하는 예제:

CI 스크립트 예제 
#!/bin/bash
# 여러 브랜치를 동시에 빌드하는 CI 스크립트

BRANCHES=("main" "develop" "staging")

for branch in "${BRANCHES[@]}"; do
  git worktree add "build-${branch}" "origin/${branch}"
  (
    cd "build-${branch}"
    npm ci && npm run build && npm test
  ) &  # 백그라운드로 병렬 실행
done

wait  # 모든 빌드 완료 대기

# 정리
for branch in "${BRANCHES[@]}"; do
  git worktree remove "build-${branch}"
done

📂 시나리오 4: Bare Repo 기반 워크플로우

깔끔한 프로젝트 구조를 위해 bare repo + worktree 조합을 사용하는 패턴:

Bare repo 워크플로우 초기 설정 
# 프로젝트 디렉토리 생성
mkdir my-project && cd my-project

# Bare 저장소 클론 (.bare 디렉토리에)
git clone --bare https://github.com/user/repo.git .bare

# .git 파일 생성 (편의를 위해)
echo "gitdir: ./.bare" > .git

# remote fetch 설정 (bare clone은 기본값이 다름)
git config remote.origin.fetch "+refs/heads/*:refs/remotes/origin/*"
git fetch origin

# 워크트리 생성
git worktree add main
git worktree add develop
git worktree add -b feature/awesome feature-awesome

# 결과 구조:
# my-project/
# ├── .bare/              ← Git object store
# ├── .git                ← gitdir 참조 파일
# ├── main/               ← main 브랜치 작업 공간
# ├── develop/            ← develop 브랜치 작업 공간
# └── feature-awesome/    ← 새 피처 작업 공간

⚠️ 주의사항과 팁

Git Worktree를 효과적으로 사용하기 위해 알아두어야 할 중요한 사항들입니다.

🚫 같은 브랜치를 동시에 체크아웃할 수 없음

이것은 가장 중요한 제약 사항입니다. 하나의 저장소에서 같은 브랜치를 두 개 이상의 워크트리에서 체크아웃할 수 없습니다.

에러 예시 
# main이 이미 메인 워크트리에서 체크아웃된 상태
$ git worktree add ../wt-main main
fatal: 'main' is already checked out at '/home/user/my-project'

# 해결책 1: 다른 브랜치 사용
git worktree add -b main-copy ../wt-main main

# 해결책 2: detached HEAD로 체크아웃
git worktree add --detach ../wt-main main
ℹ️ 왜 이런 제약이 있을까?

같은 브랜치를 두 워크트리에서 체크아웃하면, 한쪽에서 commit할 때 다른 쪽의 HEAD가 자동으로 갱신되지 않아 데이터 일관성 문제가 발생합니다. Git은 이를 원천 차단합니다.

📦 Submodule 주의점

⚠️ 주의

Submodule이 포함된 저장소에서 worktree를 생성하면, submodule이 자동으로 초기화/업데이트되지 않습니다. 새 워크트리에서 별도로 git submodule update --init을 실행해야 합니다.

Submodule이 있는 프로젝트에서의 워크트리 
# 워크트리 생성 후
git worktree add ../wt-feature feature
cd ../wt-feature

# submodule 초기화 필요!
git submodule update --init --recursive

🗑️ 올바른 정리 방법

워크트리를 정리하는 올바른 방법과 잘못된 방법을 비교합니다:

✅ 올바른 정리

  • git worktree remove 사용
  • 메타데이터와 디렉토리 동시 정리
  • 변경사항 있으면 경고 표시

❌ 잘못된 정리

  • rm -rf로 디렉토리만 삭제
  • 메타데이터가 .git/worktrees에 남음
  • git worktree prune으로 추가 정리 필요
정리 베스트 프랙티스 
# 권장: git worktree remove 사용
git worktree remove ../wt-feature

# 만약 rm -rf로 삭제했다면:
git worktree prune  # 고아 메타데이터 정리

# 전체 워크트리 상태 확인
git worktree list

💡 유용한 팁 모음

💡 워크트리 경로 네이밍 컨벤션

워크트리 디렉토리를 프로젝트 상위 디렉토리에 생성하고, wt- 접두사를 붙이면 관리하기 쉽습니다.
예: ../wt-feature, ../wt-hotfix, ../wt-review-pr123

💡 Shell 별칭 설정

자주 사용하는 worktree 명령어를 별칭으로 등록하세요:

~/.gitconfig 또는 ~/.zshrc 
# .gitconfig에 별칭 추가
[alias]
  wta = worktree add
  wtl = worktree list
  wtr = worktree remove
  wtp = worktree prune

# 또는 shell alias
alias gwta='git worktree add'
alias gwtl='git worktree list'
alias gwtr='git worktree remove'
💡 .gitignore와 워크트리

.gitignore는 저장소에 포함되므로 모든 워크트리에서 공유됩니다. 하지만 각 워크트리의 빌드 산출물이나 node_modules는 독립적입니다.

💡 IDE 통합

VSCode에서 각 워크트리를 별도 창으로 열면 완전히 독립적인 개발 환경을 구성할 수 있습니다. code ../wt-feature로 바로 열 수 있습니다.

📋 명령어 빠른 레퍼런스

모든 Git Worktree 명령어와 주요 옵션을 한눈에 확인할 수 있는 레퍼런스 테이블입니다. 검색창에서 명령어를 검색할 수도 있습니다.

명령어 설명 주요 옵션
git worktree add <path> [<branch>] 새 워크트리를 생성하고 브랜치를 체크아웃 -b -B --detach --track --lock
git worktree list 모든 워크트리의 목록을 출력 --porcelain --format
git worktree remove <path> 워크트리를 제거 (디렉토리 + 메타데이터) --force
git worktree move <path> <new-path> 워크트리를 새 경로로 이동 --force
git worktree prune 삭제된 워크트리의 고아 메타데이터 정리 --dry-run --verbose --expire
git worktree lock <path> 워크트리를 잠금 (prune 보호) --reason
git worktree unlock <path> 워크트리 잠금 해제 —
git worktree repair [<path>...] 깨진 워크트리 연결 복구 (Git 2.30+) —

🔗 관련 Git 명령어

명령어 Worktree와의 관계
git stash Worktree를 사용하면 stash 사용 빈도가 크게 줄어듦
git switch / checkout Worktree 내에서는 일반적인 브랜치 전환과 동일하게 동작
git clone --bare Bare repo + worktree 패턴의 기반
git submodule 새 워크트리에서 별도 초기화 필요
git bisect 별도 워크트리에서 bisect를 실행하면 메인 작업에 영향 없음
