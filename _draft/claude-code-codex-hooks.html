Claude Code hooks와 Codex hooks의 실행 지점, 설정 모델, 안전 경계 비교

두 제품 모두 에이전트 생명주기에 스크립트를 붙이는 확장 지점을 제공하지만, 지원 이벤트 폭과 핸들러 종류, 신뢰 절차, 차단 의미가 다릅니다. 이 문서는 공식 문서에서 확인된 사실만 비교합니다.

기준일: 2026-06-04 Claude Code 공식 hooks 문서 OpenAI Codex 공식 hooks 문서 외부 자산 없음 · inline CSS/JS

한 화면 시스템 지도

hooks는 사용자 프롬프트, 도구 실행, 권한 요청, 압축, 종료 같은 지점에서 외부 로직을 실행하는 생명주기 확장면입니다.

핵심 결론

도입 관점에서 가장 크게 달라지는 지점은 적용 범위, 실행 모델, 보안 검토 방식입니다.

Claude

이벤트 범위가 더 넓음

파일 변경, worktree, teammate idle, elicitation, 실패 이벤트 등 운영·협업·세션 주변 이벤트가 많이 노출됩니다.

Codex

신뢰와 결정성이 전면에 있음

deterministic script 실행을 중심으로 하며, non-managed command hook은 exact hash 기반 trust 절차를 거칩니다.

공통 주의

hook은 권한 경계를 대신하지 않음

특히 도구 실행 전 hook은 유용한 guardrail이지만, 공식 문서상 모든 실행 경로를 완전 차단하는 보안 경계로 취급하면 안 됩니다.

비교 매트릭스

탭으로 관점을 바꾸면 같은 사실을 도입 판단, 실행 모델, 설정 모델 순서로 볼 수 있습니다.

PermissionRequest무시됩니다.hookSpecificOutput.decision.behavior의 allow/deny로 승인 결정을 반환합니다. deny가 여러 hook 사이에서 우선합니다.exit 2는 별도 제어로 문서화되어 있지 않습니다.
PostToolUse무시됩니다.decision:"block"은 feedback을 기록하고 model-facing tool result를 대체합니다. additionalContext와 continue:false도 event별로 의미가 있습니다.exit 2는 stderr reason을 feedback으로 사용합니다. command side effect는 이미 발생했습니다.
UserPromptSubmitextra developer context로 추가됩니다.hookSpecificOutput.additionalContext를 추가하거나 decision:"block"으로 prompt를 막습니다.exit 2가 prompt를 차단하고 reason은 stderr에서 옵니다.

PreCompact / PostCompact무시됩니다.common fields를 사용하며 continue:false는 compact 전/후 흐름을 중단합니다.exit 2는 별도 제어로 문서화되어 있지 않습니다.

초보자를 위한 용어 풀이

아래 용어는 Claude Code와 Codex hook 문서에서 반복되는 실행·제어 프로토콜 표현입니다.

lifecycle / 생명주기
세션 시작, 사용자 prompt, tool 실행, compact, stop처럼 에이전트 작업이 지나가는 단계들의 흐름입니다.
hook event
그 흐름 중 hook을 실행할 수 있는 이름 붙은 시점입니다. 예: PreToolUse, PostToolUse, Stop.
matcher
어떤 hook handler를 실행할지 고르는 필터입니다. tool name, trigger, agent type 같은 값에 맞춰 선택됩니다.
stdin / stdout / stderr
command hook의 표준 입력, 표준 출력, 표준 오류입니다. 입력 JSON은 stdin으로 받고, 결정 JSON은 stdout에 쓰며, 일부 차단 이유는 stderr에서 읽습니다.
exit code
프로그램 종료 상태 번호입니다. 효과는 event별입니다. 일부 event에서는 exit code 2가 차단/continuation 신호이고, 일부 event에서는 무시되거나 별도 제어로 문서화되어 있지 않습니다.
JSON
기계가 읽기 쉬운 구조화 텍스트 형식입니다. hook 입력과 출력 결정은 대부분 JSON 객체로 표현됩니다.
hookSpecificOutput
Codex stdout JSON 안에서 event별 결정 값을 담는 영역입니다. 같은 필드라도 event마다 의미가 다릅니다.
permissionDecision
Codex PreToolUse에서 tool call을 허용하거나 거부하는 결정 값입니다. deny는 차단, allow는 허용을 뜻합니다.
additionalContext
모델에게 추가로 알려주는 설명입니다. 작업을 직접 실행하지 않고 판단에 참고할 정보를 붙이는 용도입니다.
updatedInput
지원되는 Codex PreToolUse hook에서 tool에 넘길 입력을 바꾼 값입니다. 허용 결정과 함께 써야 합니다.
continue:false
정상 흐름을 계속하지 말라는 JSON 신호입니다. 어느 event에서 멈추는지는 event별로 다릅니다.
MCP
Model Context Protocol의 약자입니다. 모델/에이전트가 외부 tool이나 서버 기능을 표준 방식으로 부르는 통로입니다.
subagent
메인 에이전트가 특정 작업을 맡기려고 만든 보조 에이전트입니다. 시작과 종료 시점에도 hook이 걸릴 수 있습니다.
compaction
긴 대화나 작업 기록을 더 짧게 압축해 다음 단계에 넘기는 과정입니다. 한국어 문서에서는 compact/압축으로도 표현했습니다.
guardrail
위험한 동작을 줄이기 위한 안전장치입니다. 단독으로 완전한 보안 차단막이라는 뜻은 아닙니다.
enforcement boundary
반드시 막아야 하는 것을 강제로 차단하는 보안 경계입니다. Codex PreToolUse는 유용한 guardrail이지만 완전한 enforcement boundary는 아닙니다.
side effect
파일 생성, 명령 실행, 외부 상태 변경처럼 이미 일어난 실제 변화입니다. post hook은 보통 side effect를 되돌리지 못합니다.
model-facing result
tool 실행 결과 중 모델이 다음 응답을 만들 때 보게 되는 내용입니다. Codex PostToolUse는 이 값을 feedback으로 바꿀 수 있습니다.
turn
사용자 입력 하나를 처리해 모델/에이전트가 응답하는 한 차례의 작업 단위입니다.
cwd
current working directory의 약자입니다. hook command가 실행되는 현재 작업 디렉터리입니다.
transcript_path
세션 대화 기록 파일의 위치입니다. 민감 정보가 섞일 수 있으므로 출력이나 공유에 주의해야 합니다.
hash trust
Codex가 command hook 내용을 해시로 식별해 사용자가 신뢰한 정확한 명령인지 확인하는 절차입니다.
parallel / concurrent
여러 hook이 같은 시점에 거의 동시에 실행될 수 있다는 뜻입니다. 같은 파일이나 상태를 같이 만질 때 충돌을 조심해야 합니다.

이벤트 표면 필터

공식 문서에서 확인된 event 이름입니다. 버튼으로 공통 이벤트와 제품별 이벤트를 좁혀 볼 수 있습니다.

현재 공식 문서 기준으로 Codex 전용 이벤트는 없습니다. Codex 이벤트 10개는 모두 Claude Code에도 존재하는 공통 이벤트로 분류됩니다.

도구 실행 주변 흐름

두 제품 모두 도구 실행 전후 지점을 다루지만, hook 결과를 보안 경계로 과신하지 않는 설계가 필요합니다.

도구 실행 흐름은 사용자 요청, PreToolUse, 도구 실행, PostToolUse 순서로 구성됩니다. Claude Code는 여러 handler 타입으로 검사할 수 있고 command hook은 사용자 시스템 권한으로 실행될 수 있습니다. Codex는 command hook trust와 동시 실행 특성이 있으며 PreToolUse를 완전한 보안 경계로 취급하지 않습니다.

안전한 예시 패턴

예시는 비파괴적이고 공개 스타일 경로만 사용합니다. 실제 배포 전에는 공식 schema와 조직 정책에 맞게 검토해야 합니다.

허용하기 좋은 작업

JSON payload 검증, 로컬 lint 명령 안내, 작업 로그 요약, 알림 메시지 생성처럼 되돌릴 수 있고 읽기 중심인 작업.

검토가 필요한 작업

shell 변수 조합, path 기반 파일 접근, 네트워크 호출, 권한 상승이 얽힌 command hook.

피해야 할 작업

임의 삭제, secret 출력, 홈 디렉터리 전체 스캔, 신뢰되지 않은 입력을 그대로 shell에 삽입하는 방식.

Claude Code: stdin payload를 읽기만 하는 command 예시
{
  "hooks": {
    "UserPromptSubmit": [
      {
        "type": "command",
        "command": "python3 -c 'import json,sys; data=json.load(sys.stdin); print(\"prompt received\")'"
      }
    ]
  }
}
Codex: stdout JSON으로 context를 추가하는 command 예시
{
  "hooks": {
    "UserPromptSubmit": [
      {
        "type": "command",
        "command": "python3 -c 'import json; print(json.dumps({\"hookSpecificOutput\":{\"hookEventName\":\"UserPromptSubmit\",\"additionalContext\":\"codex hook observed user prompt\"}}))'"
      }
    ]
  }
}

운영 체크리스트

hook은 자동화 지점이면서 동시에 로컬 실행면입니다. 배포 전 아래 항목을 정책으로 고정하는 것이 좋습니다.

보안·권한

  • command hook 입력을 JSON parser로 읽고 shell 문자열 결합을 최소화합니다.
  • path traversal, secret file 접근, 민감 파일 출력 금지를 테스트합니다.
  • Codex에서는 exact hash trust 변경 흐름을 팀 운영 절차에 포함합니다.
  • Codex exit 2 reason은 stderr에서 올 수 있으므로 민감한 입력이나 내부 경로를 stderr에 쓰지 않습니다.

신뢰성·관측

  • hook timeout, 실패 exit code, blocking 의미를 event별로 문서화합니다.
  • Codex plain stdout은 event별로 context가 되거나 무시되거나 invalid가 될 수 있으므로 JSON stdout 계약을 우선합니다.
  • Claude matching hook과 Codex matching command hook은 동시에 실행될 수 있으므로 shared state를 피하고 idempotent하게 설계합니다.
  • 도구 차단 정책은 hook 하나에 의존하지 말고 권한 설정과 함께 검증합니다.
