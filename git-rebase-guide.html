Rebase란?

git rebase는 한 브랜치의 커밋들을 다른 브랜치의 끝으로 재배치(re-base)하는 명령어입니다. 커밋의 base(기반)를 바꾼다는 의미에서 rebase라고 부릅니다.

Merge가 두 브랜치의 히스토리를 합치면서 병합 커밋을 만드는 것과 달리, rebase는 커밋들을 새로운 베이스 위에 하나씩 다시 적용하여 마치 처음부터 그 위에서 작업한 것처럼 히스토리를 만듭니다.

핵심 원리: Rebase는 기존 커밋을 수정하는 것이 아니라, 동일한 변경사항을 가진 새로운 커밋을 생성합니다. 따라서 커밋 해시(SHA)가 변경됩니다.

Merge vs Rebase 비교

Merge

Before merge: A───B───C (main) ╲ D───E (feature) After git merge feature: A───B───C───M (main) ╲ ╱ D───E (feature)
  • 병합 커밋 M이 생성됨
  • 두 브랜치의 히스토리가 모두 보존
  • 비파괴적 — 기존 커밋 변경 없음
  • 히스토리가 복잡해질 수 있음

Rebase

Before rebase: A───B───C (main) ╲ D───E (feature) After git rebase main (on feature): A───B───C───D'──E' (feature)
  • 병합 커밋 없이 선형 히스토리
  • 커밋이 새로 생성됨 (D→D', E→E')
  • 깔끔한 히스토리 — 읽기 쉬움
  • 공유된 커밋을 rebase하면 위험

언제 어떤 것을 쓸까?

상황추천이유
로컬 feature 브랜치를 최신 main에 맞추기 Rebase 깔끔한 히스토리 유지, 아직 공유되지 않은 커밋
공유된 브랜치에 feature 통합 Merge 이미 push된 커밋의 히스토리 보존
PR 전 커밋 정리 Rebase -i 리뷰어가 읽기 좋은 커밋 단위로 정리
release 브랜치 통합 Merge 통합 시점을 명확히 기록

기본 Rebase

현재 브랜치의 커밋들을 대상 브랜치 위로 재배치합니다.

# feature 브랜치에서 main의 최신 커밋 위로 rebase
git checkout feature
git rebase main

# 또는 한 줄로
git rebase main feature

워크플로우

1main 브랜치에서 feature 브랜치 생성 후 작업
2main에 새 커밋이 추가됨 (다른 사람의 작업)
3feature 브랜치에서 git rebase main 실행
4feature의 커밋들이 main 최신 커밋 뒤에 재배치됨
5main으로 돌아가서 git merge feature → fast-forward 병합
# 전체 워크플로우 예시
git checkout feature
git rebase main
git checkout main
git merge feature     # fast-forward!

git rebase --onto

특정 범위의 커밋만 다른 베이스로 옮길 때 사용합니다. 세 개의 인자를 받습니다.

git rebase --onto <newbase> <oldbase> <branch>

<oldbase>...<branch> 범위의 커밋들을 <newbase> 위로 옮깁니다.

시나리오: feature에서 갈라진 sub-feature를 main 위로 직접 옮기기

Before: A───B───C (main) ╲ D───E (feature) ╲ F───G (sub-feature) git rebase --onto main feature sub-feature After: A───B───C───F'──G' (sub-feature) ╲ D───E (feature)

시나리오: 특정 범위의 커밋만 옮기기

# feature의 최근 3개 커밋만 main 위로 옮기기
git rebase --onto main feature~3 feature

Upstream Rebase (fetch + rebase)

원격의 최신 변경사항을 가져오면서 rebase하는 방법입니다.

# 방법 1: fetch 후 rebase
git fetch origin
git rebase origin/main

# 방법 2: pull --rebase (위와 동일)
git pull --rebase origin main

# 기본 pull 동작을 rebase로 설정
git config --global pull.rebase true

Interactive Rebase란?

git rebase -i는 rebase 과정에서 각 커밋에 대해 어떤 작업을 할지 선택할 수 있게 해줍니다. 커밋 합치기, 메시지 수정, 순서 변경, 삭제 등을 자유롭게 할 수 있습니다. 

# 최근 N개 커밋을 interactive rebase
git rebase -i HEAD~N

# 특정 커밋부터 interactive rebase
git rebase -i <commit-hash>

# main 브랜치 이후의 모든 커밋을 대상으로
git rebase -i main

편집기에 표시되는 형식

pick a1b2c3d 첫 번째 커밋 메시지
pick e4f5g6h 두 번째 커밋 메시지
pick i7j8k9l 세 번째 커밋 메시지

# Rebase a1b2c3d..i7j8k9l onto 0f0f0f0 (3 commands)
#
# Commands:
# p, pick   = use commit
# r, reword = use commit, but edit the commit message
# e, edit   = use commit, but stop for amending
# s, squash = use commit, but meld into previous commit
# f, fixup  = like "squash", but discard this commit's log message
# d, drop   = remove commit
커밋은 위에서 아래로 (오래된 것 → 새로운 것) 나열됩니다. git log와 반대 순서입니다.

명령어 옵션 상세

명령어단축키설명
pickp커밋을 그대로 사용 (기본값)
rewordr커밋은 유지하되, 커밋 메시지를 수정
edite커밋에서 멈추어 내용을 수정 가능 (amend)
squashs이전 커밋과 합치며, 메시지도 합침
fixupf이전 커밋과 합치되, 이 커밋의 메시지는 버림
dropd커밋을 삭제

pick 커밋 그대로 사용

▼

기본 동작입니다. 커밋을 변경 없이 그대로 적용합니다. 커밋 순서를 바꾸고 싶다면 줄의 순서를 변경하면 됩니다.

# 순서 변경 예시 — 커밋 순서를 바꿈
pick i7j8k9l 세 번째 커밋  ← 위로 이동
pick a1b2c3d 첫 번째 커밋
pick e4f5g6h 두 번째 커밋

reword 커밋 메시지 수정

▼

커밋의 코드 변경은 그대로 두고 메시지만 수정합니다. 저장 후 편집기가 다시 열리며 새 메시지를 입력합니다.

reword a1b2c3d 오타가 있는 커밋 메세지  ← 편집기에서 수정 가능
pick e4f5g6h 두 번째 커밋
pick i7j8k9l 세 번째 커밋
여기서 메시지를 바꿔도 무시됩니다. 저장 후 열리는 별도 편집기에서 수정하세요.

edit 커밋 수정 후 계속

▼

해당 커밋에서 rebase를 일시 정지합니다. 파일을 수정하거나 커밋을 분할할 수 있습니다.

edit a1b2c3d 수정이 필요한 커밋
pick e4f5g6h 두 번째 커밋

편집기를 저장하면 해당 커밋에서 멈춥니다. 이후:

# 파일 수정 후
git add .
git commit --amend     # 현재 커밋 수정
git rebase --continue  # 나머지 rebase 계속

# 또는 커밋을 분할하려면
git reset HEAD^          # 커밋을 해제 (변경사항은 유지)
git add file1.js
git commit -m "파트 1"
git add file2.js
git commit -m "파트 2"
git rebase --continue

squash 커밋 합치기 (메시지 포함)

▼

바로 위의 커밋과 합칩니다. 두 커밋의 메시지를 합쳐서 편집할 수 있는 편집기가 열립니다.

pick   a1b2c3d 기능 구현
squash e4f5g6h 오타 수정      ← 위의 커밋에 합쳐짐
squash i7j8k9l 리뷰 반영      ← 위의 커밋에 합쳐짐

결과: 3개의 커밋이 1개로 합쳐지며, 3개의 메시지를 편집할 수 있습니다.

첫 번째 커밋은 squash일 수 없습니다. 합칠 대상(위의 커밋)이 필요합니다.

fixup 커밋 합치기 (메시지 버림)

▼

squash와 같지만 합쳐지는 커밋의 메시지를 버립니다. 빠르게 자잘한 수정을 합칠 때 유용합니다.

pick  a1b2c3d 로그인 기능 구현
fixup e4f5g6h lint 오류 수정     ← 메시지 버려짐
fixup i7j8k9l 빠뜨린 파일 추가   ← 메시지 버려짐

결과: 커밋 메시지는 "로그인 기능 구현"만 남습니다.

팁: git commit --fixup=<hash>와 git rebase -i --autosquash를 함께 쓰면 fixup 대상을 자동으로 배치해줍니다.

drop 커밋 삭제

▼

해당 커밋을 히스토리에서 완전히 제거합니다. 줄을 삭제하는 것과 동일한 효과입니다.

pick a1b2c3d 필요한 커밋
drop e4f5g6h 디버그 코드 (삭제)   ← 제거됨
pick i7j8k9l 필요한 커밋
주의: 삭제된 커밋의 변경사항이 이후 커밋에서 참조된다면 충돌이 발생할 수 있습니다.

Rebase 중 충돌 발생

Rebase는 커밋을 하나씩 적용하므로, 각 커밋마다 충돌이 발생할 수 있습니다. 충돌이 발생하면 rebase가 멈추고 수동 해결을 기다립니다.

CONFLICT (content): Merge conflict in src/app.js
error: could not apply e4f5g6h... 두 번째 커밋
hint: Resolve all conflicts manually, mark them as resolved with
hint: "git add/rm <conflicted_files>", then run "git rebase --continue".
hint: You can instead skip this commit: "git rebase --skip".
hint: To abort and get back to the state before "git rebase", run
hint: "git rebase --abort".

충돌 해결 흐름

충돌 발생! │ ├──→ 해결하고 계속 ─── git rebase --continue │ ├──→ 이 커밋 건너뛰기 ── git rebase --skip │ └──→ rebase 전체 취소 ── git rebase --abort

해결하고 계속하기 (--continue)

가장 일반적인 방법입니다. 충돌을 수동으로 해결한 후 rebase를 계속합니다.

1충돌 파일 확인: git status
Unmerged paths:
  both modified:   src/app.js
2충돌 파일을 열어 수동 해결
<<<<<<< HEAD
console.log("main 브랜치의 코드");
=======
console.log("feature 브랜치의 코드");
>>>>>>> e4f5g6h (두 번째 커밋)

↓ 원하는 코드로 수정

console.log("병합된 최종 코드");
3해결된 파일을 staging
git add src/app.js
4rebase 계속
git rebase --continue
남은 커밋에서 또 충돌이 나면 같은 과정을 반복합니다. 모든 커밋이 적용될 때까지 계속됩니다.

커밋 건너뛰기 (--skip)

현재 충돌이 발생한 커밋을 완전히 건너뛰고 다음 커밋으로 넘어갑니다.

git rebase --skip
주의: 해당 커밋의 모든 변경사항이 사라집니다. 이미 다른 곳에 반영된 중복 커밋인 경우에만 사용하세요.

Rebase 전체 취소 (--abort)

rebase 시작 전 상태로 완전히 되돌립니다. 안전한 탈출구입니다.

git rebase --abort

모든 변경사항이 취소되고, 브랜치는 rebase 시작 전 상태로 복원됩니다.

팁: 복잡한 충돌이 예상되면 rebase 전에 백업 브랜치를 만들어 두세요:
git branch backup-feature feature

유용한 충돌 해결 도구

# 충돌 상태 확인
git status

# 3-way merge 도구 사용
git mergetool

# 충돌 시 한쪽을 통째로 선택
git checkout --ours file.js    # rebase 대상 (base) 선택
git checkout --theirs file.js  # 현재 브랜치 커밋 선택

# 현재 rebase 진행 상황 확인
ls .git/rebase-merge/
ours/theirs 주의: Rebase에서는 merge와 반대입니다!
--ours = rebase 대상 브랜치 (main)
--theirs = 현재 적용 중인 커밋 (feature의 커밋)

The Golden Rule of Rebase

이미 공유된(push된) 커밋은 절대 rebase하지 마라.

Rebase는 새로운 커밋을 생성합니다. 이미 다른 사람이 가져간 커밋을 rebase하면, 같은 변경사항을 가진 다른 커밋이 두 벌 존재하게 되어 히스토리가 꼬입니다.

위험한 시나리오: 팀원 A가 push한 커밋: A───B───C───D───E (origin/feature) 팀원 B가 같은 커밋을 rebase: A───B───C───D'──E' (feature, rebased) 팀원 B가 force push 후, 팀원 A가 pull하면: D, E와 D', E'가 중복 → 히스토리 혼란!

위험 Rebase 하면 안 되는 상황

  • 공유 브랜치 — main, develop 등 여러 사람이 사용하는 브랜치
  • 이미 push한 커밋 — 다른 사람이 해당 커밋을 기반으로 작업 중일 수 있음
  • merge 커밋이 포함된 히스토리 — 복잡한 충돌이 반복 발생할 수 있음
  • 다른 브랜치의 베이스가 되는 커밋 — 파생 브랜치들이 모두 영향받음

안전 Rebase 해도 되는 상황

  • 로컬 전용 커밋 — 아직 push하지 않은 나만의 커밋
  • 개인 feature 브랜치 — 혼자만 사용하는 브랜치 (force push 가능)
  • PR 전 커밋 정리 — 아직 merge 되지 않은 PR의 커밋 정리
  • 팀 내 합의된 경우 — 모두가 rebase 워크플로우에 동의한 경우

주의 안전한 사용 가이드라인

  1. rebase 전에 백업 브랜치 생성 
    git branch backup/feature-$(date +%Y%m%d)
  2. force push 시 --force-with-lease 사용 
    git push --force-with-lease origin feature

    --force와 달리 원격에 내가 모르는 새 커밋이 있으면 push를 거부합니다.

  3. ORIG_HEAD로 되돌리기 
    # rebase 직후라면 이전 상태로 복구 가능
git reset --hard ORIG_HEAD
  4. reflog으로 언제든 복구 
    git reflog
# 원하는 시점의 해시를 찾아서
git reset --hard HEAD@{N}

판단 플로우차트

이 커밋을 rebase 해도 될까? │ ├─ 이 커밋을 push 한 적이 있는가? │ ├─ 아니오 → ✅ 안전하게 rebase 가능 │ └─ 예 │ │ │ ├─ 나만 쓰는 브랜치인가? │ │ ├─ 예 → ⚠️ force-with-lease로 push 가능 │ │ └─ 아니오 │ │ │ │ │ └─ 팀과 합의했는가? │ │ ├─ 예 → ⚠️ 팀원에게 알린 후 진행 │ │ └─ 아니오 → ❌ rebase 하지 마세요!

레시피 1 최근 커밋 N개 합치기 (Squash)

▼

자잘한 커밋들을 하나의 의미 있는 커밋으로 합칩니다.

# 최근 3개 커밋을 하나로
git rebase -i HEAD~3

# 편집기에서:
pick   a1b2c3d 기능 구현
fixup  e4f5g6h typo 수정
fixup  i7j8k9l 빠진 import 추가
더 빠른 방법: git reset --soft HEAD~3 && git commit으로도 동일한 결과를 얻을 수 있습니다.

레시피 2 커밋 순서 바꾸기

▼

편집기에서 줄 순서를 변경하면 커밋 순서가 바뀝니다.

git rebase -i HEAD~3

# 변경 전:
pick a1b2c3d A 기능
pick e4f5g6h B 기능
pick i7j8k9l C 기능

# 변경 후 (B와 C 순서 교환):
pick a1b2c3d A 기능
pick i7j8k9l C 기능  ← 위로
pick e4f5g6h B 기능  ← 아래로
커밋 간 의존성이 있으면 충돌이 발생할 수 있습니다.

레시피 3 하나의 커밋을 여러 개로 분할

▼

너무 큰 커밋을 논리적 단위로 분할합니다.

git rebase -i HEAD~2

# 분할할 커밋을 edit으로 변경:
edit a1b2c3d 큰 커밋 (분할 필요)
pick e4f5g6h 다른 커밋

저장 후 멈추면:

# 커밋을 해제 (변경사항은 working directory에 유지)
git reset HEAD^

# 논리적 단위로 나누어 커밋
git add src/api/*
git commit -m "API 엔드포인트 추가"

git add src/ui/*
git commit -m "UI 컴포넌트 추가"

레시피 4 특정 커밋만 다른 브랜치에서 가져오기

▼

Cherry-pick과 유사하지만 rebase --onto로도 가능합니다.

# cherry-pick: 가장 간단한 방법
git cherry-pick <commit-hash>

# rebase --onto: 연속된 범위의 커밋을 가져올 때
# feature의 C~E 커밋만 main 위로
git rebase --onto main C^ E

레시피 5 커밋 메시지 일괄 수정

▼

여러 커밋의 메시지를 한 번에 수정합니다.

git rebase -i HEAD~4

# 수정할 커밋을 reword로 변경:
reword a1b2c3d old msg 1
pick   e4f5g6h ok msg
reword i7j8k9l old msg 2
reword m0n1o2p old msg 3

저장 후, reword로 표시한 각 커밋마다 편집기가 열립니다.

레시피 6 autosquash로 자동 fixup

▼

작업 중 발견한 수정사항을 나중에 자동으로 합칠 수 있습니다.

# 1. 수정사항을 fixup 커밋으로 생성
git commit --fixup=<target-hash>

# 이러면 "fixup! 원래 커밋 메시지" 형태의 커밋이 생깁니다

# 2. autosquash로 rebase하면 자동 배치
git rebase -i --autosquash main

# 편집기에서 fixup 커밋이 자동으로 올바른 위치에 배치됨!
pick  a1b2c3d 원래 커밋
fixup x9y8z7w fixup! 원래 커밋  ← 자동 배치
pick  e4f5g6h 다른 커밋
영구 설정: git config --global rebase.autosquash true로 항상 autosquash를 활성화할 수 있습니다.

레시피 7 잘못된 rebase 되돌리기

▼

rebase를 잘못했을 때 복구하는 방법들입니다.

# 방법 1: ORIG_HEAD 사용 (rebase 직후에만)
git reset --hard ORIG_HEAD

# 방법 2: reflog에서 rebase 이전 상태 찾기
git reflog
a1b2c3d HEAD@{0}: rebase (finish): ...
e4f5g6h HEAD@{1}: rebase (start): ...
f7g8h9i HEAD@{2}: commit: 마지막 정상 커밋

git reset --hard HEAD@{2}

# 방법 3: 백업 브랜치에서 복구
git reset --hard backup/feature
reflog은 로컬 Git의 모든 HEAD 변경 기록을 보관합니다. 기본 90일간 유지되므로, 거의 모든 실수를 복구할 수 있습니다.

레시피 8 Pull 시 항상 rebase 사용

▼

불필요한 merge 커밋을 방지하기 위해 pull 기본 동작을 rebase로 설정합니다.

# 전역 설정
git config --global pull.rebase true

# 특정 저장소만
git config pull.rebase true

# 일회성 사용
git pull --rebase

# stash를 자동으로 처리 (uncommitted 변경사항이 있을 때)
git config --global rebase.autoStash true
