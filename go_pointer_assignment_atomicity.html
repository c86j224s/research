p = q 한 줄이 정말 안전한가? Go Memory Model이 말하는 진실

핵심 요약

TL;DR — 세 가지 핵심

🚫

포인터 대입은 Atomic이 아니다

Go 언어 스펙 차원에서 p = q 같은 평범한 포인터 대입은 atomic operation으로 보장되지 않습니다. 동기화 없이 한 goroutine에서 쓰고 다른 goroutine에서 읽으면 data race이며, 미정의 동작을 초래합니다.

⚠️

Torn Read 방지 ≠ 가시성 보장

단일 워드 크기 읽기에서 찢어진 값은 보이지 않지만, 이는 happens-before 보장이 아닙니다. 포인터를 통해 읽은 구조체 필드들이 일관되게 보인다는 보장은 전혀 없습니다.

✅

sync/atomic을 반드시 사용

atomic.Pointer[T] (Go 1.19+), atomic.Value, 또는 mutex/channel을 반드시 사용해야 합니다. 이들은 데이터 레이스를 제거하고 happens-before 엣지를 만들어 줍니다.

Go Memory Model 공식 예시

왜 위험한가 — 코드로 보기

각 코드 줄을 클릭하면 해당 줄의 위험성을 확인할 수 있습니다.

Go 
type T struct{ msg string }
var g *T
 
func setup() {
    t := new(T)
    t.msg = "hello, world"
    g = t
}
 
func main() {
    go setup()
    for g == nil {}
    print(g.msg)
}

Go Memory Model 공식 경고: "Even if main observes g != nil and exits its loop, there is no guarantee that it will observe the initialized value for g.msg."

두 가지 의미의 "Atomic"

하드웨어 보장 vs 언어 보장

Dave Cheney가 구분한 두 가지 다른 의미의 "atomic". 각 레이어를 클릭하면 상세 설명을 볼 수 있습니다.

하드웨어 레이어

인스트럭션 차원의 Atomicity (1)

정렬된 워드 크기 load/store는 찢어지지 않는다 — x86-64, ARMv8 모두 보장

x86-64에서 정렬된 8바이트 store는 단일 인스트럭션으로 실행되어 torn write가 발생하지 않습니다. ARMv8 등 약한 메모리 모델에서도 정렬된 워드 단위의 atomicity 자체는 보장됩니다.

그러나 — 이는 다른 코어에 그 결과가 언제 가시화되는지, 주변 메모리 연산이 재정렬되지 않는지에 대해서는 아무 말도 하지 않습니다.

추가 위험: 32-bit ARM, 32-bit MIPS에서는 64-bit 값(int64/uint64)이 8-byte 정렬되어야 atomic이 보장되며, 과거에는 호출자 책임이었습니다.

≠ 같지 않다!
언어/sync/atomic 레이어

동기화(Synchronizing) Atomicity (2)

다른 코어/goroutine에 즉각 가시화 + 주변 코드 재정렬 차단 — sync/atomic만이 보장

sync/atomic이 보장하는 것은 sequentially-consistent atomics입니다. C++의 memory_order_seq_cst, Java의 volatile과 동등합니다.

이 연산은 메모리 펜스(memory barrier)를 삽입하여 atomic 연산 전후의 일반 read/write가 그 경계를 넘어 이동하지 못하게 합니다.

평범한 포인터 대입은 (1)도 언어 차원에서는 약속되지 않았습니다. 하드웨어 보장에 의존하면 약한 모델(ARM, POWER 등)로 이식할 때 깨지며, 컴파일러 최적화는 하드웨어와 별개로 동기화 없는 코드를 마음대로 변형할 수 있습니다.

올바른 해결책

atomic.Pointer[T] 사용법

atomic.Pointer[T]를 사용해야 하는 5가지 이유:

1 데이터 레이스 자체를 제거 ▼
Memory Model의 정의상 "모든 접근이 sync/atomic의 atomic 데이터 접근이라면" race가 아닙니다. race detector(-race)도 이 경우만 통과시킵니다.
2 Sequentially-consistent happens-before 보장 ▼
갱신된 Memory Model: "If the effect of an atomic operation A is observed by atomic operation B, then A is synchronized before B." C++의 memory_order_seq_cst 및 Java의 volatile과 동등합니다.
3 포인터가 가리키는 객체의 publication 보장 ▼
한 goroutine이 새 구조체의 모든 필드를 채운 뒤 p.Store(newPtr)을 호출하면, 다른 goroutine이 p.Load()로 그 포인터를 받았을 때 모든 필드를 일관되게 보게 됩니다. 평범한 대입에서는 이런 보장이 없습니다.
4 컴파일러·CPU 재정렬 차단 ▼
메모리 펜스(memory barrier)를 삽입하여 atomic 연산 주위의 일반 read/write가 그 너머로 이동하지 못하게 합니다. 평범한 포인터 대입에는 그런 차단이 없으며, 컴파일러는 단일 스레드 의미가 보존되는 한 자유롭게 재배치할 수 있습니다.
5 타입 안전성 + unsafe.Pointer 회피 ▼
atomic.Value는 interface{}를 받기 때문에 타입 단언이 필요하고, 옛 LoadPointer/StorePointer는 unsafe.Pointer를 요구했습니다. 제네릭 기반 atomic.Pointer[T]는 이 둘을 해결합니다.

Store → happens-before → Load 플로우

Writer Goroutine
settings.Store(cfg)
happens-before
Reader Goroutine
settings.Load()

Load()가 Store()의 결과를 관찰하면, Store() 이전의 모든 쓰기가 Load() 이후에 가시적

Go — Read-heavy 설정 핫스왑 예시 
type Settings struct{ Port int; Host string }
 
var settings atomic.Pointer[Settings]
 
// 갱신 측 (writer)
func update(p int, h string) {
    cfg := &Settings{Port: p, Host: h}
    settings.Store(cfg)  // happens-before 엣지 생성
}
 
// 다수의 reader — lock-free
func current() *Settings { return settings.Load() }
최적 사용 사례

read-heavy + 가끔 write: 설정 핫리로드, 라우팅 테이블, copy-on-write 자료구조, 캐시된 컴파일 객체

성능 이점

Dmitry Vyukov: 80코어 머신에서 encoding/gob의 mutex → atomic 전환 시 16코어에서 2배, 80코어에서 ~20배 성능 향상 가능

mutex가 나은 경우

여러 필드의 일관된 갱신이 필요할 때는 mutex가 더 자연스러움 — atomic은 단일 위치만 보호

실제 문제

동기화 없는 포인터 대입의 위험

🔴 Critical 인터페이스/슬라이스 메모리 Corruption ▼
interface{}, []T, string, map 등은 다중 워드 값입니다. 이런 값에 대한 race는 (포인터, 타입) 또는 (포인터, 길이) 쌍이 어긋나면서 타입 안전성을 깨뜨려 임의의 메모리 corruption이나 보안 취약점으로 이어질 수 있습니다. Dmitry Vyukov가 GoogleCTF 2019 finals의 gomium pwn challenge에서 시연한 바 있습니다.
🔴 Critical 부분 초기화된 객체 노출 ▼
새 구조체를 채운 후 포인터를 대입했더라도, 다른 goroutine이 새 포인터를 보면서 옛 필드 값이나 일부만 초기화된 상태를 볼 수 있습니다. nil pointer dereference panic으로 이어지기도 합니다. Go 블로그의 race detector 소개에서 timer 예시가 정확히 이 부류입니다.
🟡 Warning Race Detector 경고 ▼
한 goroutine의 l.c = newClient와 다른 goroutine의 return l.c는 즉시 WARNING: DATA RACE를 띄웁니다. -race 플래그로 빌드할 때 보고되며, 프로덕션에서는 프로세스를 종료시킬 수도 있습니다.
🟡 Warning Stale View — 무한 루프 ▼
컴파일러가 루프 안의 for g == nil {} 같은 코드를 "g는 이 루프 안에서 변하지 않는다"고 판단해 hoisting 하여 무한 루프가 끝나지 않을 수 있습니다. 이는 Go Memory Model이 직접 예시로 설명하는 패턴입니다.
🔵 Info GC와의 상호작용 ▼
포인터의 일부만 갱신된 상태가 GC에 노출되면 dangling pointer로 이어질 수 있어 Go 런타임이 write barrier를 사용합니다. 그러나 이는 race-free 프로그램을 가정한 설계이므로 동기화 없는 포인터 대입은 GC의 정확성도 위협합니다.

잘못된 흔한 가정들

미신 vs 사실

❌ 미신
"x86이니까 어차피 atomic 아닌가?"
✅ 사실
명령어 차원에서 정렬된 워드 store는 찢어지지 않지만, 컴파일러가 그 store를 다른 메모리 연산과 재배치할 수 있고, 다른 코어의 store buffer가 즉시 visibility를 보장하지 않습니다. ARM/POWER로 이식하면 깨집니다.
❌ 미신
"정렬되어 있고 8바이트니 race detector도 너그럽지 않나?"
✅ 사실
Race detector는 Go Memory Model을 구현합니다. 하드웨어 의미가 아닙니다. 동기화가 없으면 워드 크기와 무관하게 race로 보고합니다.
❌ 미신
"Go에는 volatile이 없으니 자동으로 처리되겠지?"
✅ 사실
Go에는 volatile 키워드가 없습니다. 동기화는 sync, sync/atomic, channel 중 하나로 명시적으로 수행해야 합니다.
❌ 미신
"구조체 자체는 안 바뀌고 포인터만 바뀌니 대입만 하면 되겠지?"
✅ 사실
새 구조체의 필드 초기화 → 포인터 대입의 순서가 다른 goroutine 시점에서 보존된다는 보장이 없습니다. publishing을 위한 atomic store가 필요합니다.
코드 비교

권장 패턴 vs 안티패턴

❌ 안티패턴 — 동기화 없는 포인터 대입
var g *T
 
// writer goroutine
t := new(T)
t.msg = "hello"
g = t  // ⚠️ data race!
 
// reader goroutine
if g != nil {
    print(g.msg) // 💥
}
✅ 권장 패턴 — atomic.Pointer[T]
var g atomic.Pointer[T]
 
// writer goroutine
t := new(T)
t.msg = "hello"
g.Store(t) // ✅ safe
 
// reader goroutine
if p := g.Load(); p != nil {
    print(p.msg) // ✅ 일관됨
}
주의사항

알아둬야 할 것들

Race Detector의 한계

ThreadSanitizer 기반 동적 분석이라 실제 실행된 코드 경로의 race만 잡습니다. 테스트가 동시성을 충분히 커버하지 않으면 race를 놓칠 수 있습니다.

단일 워드 읽기 ≠ happens-before

"race가 있어도 워드 단위 read는 어떤 실제 write 값을 본다"는 약속은 가시성 시점이나 주변 메모리의 일관성에 대해서는 아무것도 말하지 않습니다.

atomic.Value의 타입 제약

첫 Store 이후 같은 concrete 타입만 받습니다. 다른 타입을 넣으면 panic. nil interface도 Store하면 panic합니다. Go 1.19+에서는 atomic.Pointer[T]를 쓰세요.

"Don't be clever"

Go 철학: atomic이나 unsafe로 lock-free 코드를 짜기보다 channel과 mutex로 동기화. atomic은 정말 필요할 때만 쓰는 도구로 위치합니다.

하드웨어 ≠ 언어 보장

x86-64에서 정렬된 포인터 store는 거의 항상 찢어지지 않지만, 이는 언어 보장이 아닙니다. 32-bit 플랫폼에서는 8-byte 값에 대해 정렬 자체가 자동 보장되지 않습니다.

비공식 블로그 주의

"x86에서는 atomic"이라고 단순화하는 비공식 글들이 있지만, 공식 Go Memory Model과 Go 팀 멤버들의 발언은 일관되게 "언어 차원에서는 보장하지 않는다"입니다.

결론

var p *MyStruct = q 같은 평범한 포인터 대입은 Go 언어 스펙과 Go Memory Model 어디에서도 atomic operation으로 보장되지 않습니다. 동시 접근이 일어난다면 그 자체로 데이터 레이스이며, 정의상 정확한 동작을 약속받을 수 없습니다.

동시성 환경에서 포인터를 안전하게 공유·교체하려면 sync/atomic.Pointer[T] (Go 1.19+) 또는 atomic.Value, mutex, channel 중 하나를 반드시 사용해야 합니다.
