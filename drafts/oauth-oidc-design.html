🔑 서명 키 관리
키 순환 전략
  • RSA 2048 또는 EC P-256 키쌍을 DB에 PEM으로 저장 (암호화 권장)
  • 각 키에 kid (UUID), created_at, expires_at, status (active/retired) 관리
  • JWT 발급 시 현재 active 키 사용, kid를 Header에 포함
  • JWKS 엔드포인트에 active + 아직 유효한 retired 키 모두 노출 (전환 기간)
  • 키 순환: 새 키 생성 → active로 전환 → 이전 키 retired로 → 이전 키로 발급된 토큰 만료 후 JWKS에서 제거
💡
초기 구현 권장: RS256 단일 키
처음엔 시작 시 RSA 키쌍을 환경변수/파일에서 로드하는 방식으로 단순하게 시작. 키 순환은 Phase 2에서 DB 저장 방식으로 확장.

🔗 Federated Login — Google 연동 설계
ℹ️
우리 AS의 이중 역할
Google에 대해서는 RP(Relying Party) — Google의 ID Token을 검증하고 사용자를 식별합니다.
우리 Client들에게는 OP(OpenID Provider) — 평소처럼 우리 Access Token / ID Token을 발급합니다.
Client 입장에서는 사용자가 Google로 로그인했는지 알 필요가 없습니다.
Google 연동 플로우
계정 연결 시나리오별 처리
시나리오처리주의사항
최초 Google 로그인 (기존 계정 없음) users 신규 생성 (password_hash=NULL) + external_identities 연결 email_verified는 Google의 email_verified 값 사용
기존 이메일+PW 계정과 같은 이메일로 Google 로그인 기존 user에 external_identities만 추가 (자동 연결) Google이 email_verified=true인 경우에만 자동 연결. 아니면 별도 확인 필요.
이미 연결된 Google 계정으로 재로그인 external_identities → user_id 조회, 토큰 발급 —
계정 연결 해제 external_identities 행 삭제. password_hash도 NULL이면 로그인 불가 → 먼저 비밀번호 설정 유도 마지막 credential 해제 방지 필수
Google 계정으로 만든 계정에 비밀번호 추가 users.password_hash 업데이트 (비밀번호 설정 플로우) —
Google OAuth 앱 등록 시 필요한 것
  • Google Cloud Console에서 OAuth 2.0 Client ID 생성 (Web application 타입)
  • Authorized redirect URIs: https://auth.example.com/oauth/callback/google
  • 환경변수로 관리: GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET
  • 요청 scope: openid email profile
  • 우리 AS도 Google에 대해 PKCE 사용 (S256)
  • Google Discovery: https://accounts.google.com/.well-known/openid-configuration

🔐 OAuth 2.0 엔드포인트
GET /authorize Authorization Code 발급 시작 — 사용자 인증 및 Consent
Query Parameters
파라미터필수설명
response_typeMUST항상 code
client_idMUST등록된 Client ID
redirect_uriSHOULD등록된 URI와 exact match. 하나만 등록된 경우 생략 가능.
scopeSHOULD공백 구분. OIDC는 반드시 openid 포함. e.g. openid profile email
state보안상 MUSTCSRF 방지 불투명 값. Client가 생성 후 callback에서 검증.
code_challengeMUSTPKCE: BASE64URL(SHA256(code_verifier))
code_challenge_methodMUST반드시 S256. plain 미지원.
nonceOIDC 권장ID Token replay 방지. Client가 생성.
prompt선택none · login · consent · select_account
max_age선택인증 후 허용 최대 경과 시간(초). 초과 시 재인증.
login_hint선택로그인 폼에 미리 채울 이메일
acr_values선택요구 인증 수준 e.g. mfa
처리 흐름
  1. 파라미터 검증 (client_id 유효, redirect_uri 일치, response_type=code, code_challenge_method=S256)
  2. 로그인 세션 확인 → 없거나 max_age 초과 → /login으로 redirect (state 보존)
  3. prompt=login → 세션 있어도 강제 재인증
  4. acr_values=mfa 요구 시 세션에 MFA 완료 여부 확인 → 미완료면 /totp으로 redirect
  5. Consent 확인 → require_consent=true이고 기존 동의 없으면 /consent으로 redirect
  6. Authorization Code 생성 (crypto/rand 32바이트 hex) → Redis 저장 (TTL 2분)
  7. redirect_uri로 302 redirect: ?code={code}&state={state}
성공 응답 (302 Redirect)
HTTP/1.1 302 Found
Location: https://app.example.com/callback?code=a1b2c3d4e5...&state=xyz
에러 응답 (redirect_uri로 전달)
HTTP/1.1 302 Found
Location: https://app.example.com/callback?error=access_denied&error_description=User+denied+consent&state=xyz

// redirect_uri 자체가 유효하지 않으면 → redirect 없이 400 HTML 에러 페이지

Content-Type

Phase 1 — 기반 인프라
  • PostgreSQL 스키마 마이그레이션 (golang-migrate)P1
  • Redis 연결 및 키 TTL 관리 유틸P1
  • 환경변수 기반 설정 (issuer URL, DB URL, Redis URL, 키 경로)P1
  • 구조화된 로거 (slog 또는 zap)P1
  • HTTP 미들웨어: 요청 ID, 로깅, recover(패닉 방지)P1
  • X-Forwarded-Proto/For 헤더 파싱 (신뢰할 프록시 IP 화이트리스트)P1
  • RSA 키쌍 로드 및 JWKS 생성 유틸 (lestrrat-go/jwx)P1
Phase 2 — 사용자 인증
  • 사용자 생성/조회 (users 테이블)P2
  • 비밀번호 해시: argon2id 또는 bcrypt(cost=12)P2
  • 로그인 처리: 이메일+비밀번호 검증P2
  • 로그인 Rate Limiting (IP+email, Redis INCR+TTL)P2
  • 계정 잠금 (5회 실패 → DB locked_until 설정)P2
  • 세션 발급/검증/삭제 (Redis, HttpOnly 쿠키)P2
  • CSRF 토큰 (Double Submit Cookie 패턴)P2
  • GET/POST /login HTML 페이지P2
  • 감사 로그: login.success, login.failP2
Phase 3 — TOTP
  • TOTP 시크릿 생성 + otpauth:// URI + QR코드 (pquerna/otp)P3
  • TOTP 등록 플로우 (확인 코드 입력 후 활성화)P3
  • TOTP 검증: ±1 step 허용, replay 방지 (Redis TTL 90초)P3
  • 복구 코드 8개 생성 (bcrypt 해시 저장, 사용 시 삭제)P3

Phase 4 — Authorization Code + PKCE
