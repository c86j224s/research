Fleet은 정확도를 운영체계로 만든다. Plasma는 어디를 배울 것인가.

Fleet은 목표, 근거, 역할, 산출물을 단계마다 검문해 에이전트의 실수를 줄인다. Plasma는 source/read-first 경계를 통해 연구 보고의 근거성을 지킨다. 이 보고서는 두 접근을 같은 축에 놓고, Plasma가 가져갈 만한 작은 장치를 우선순위로 정리한다.

Fleet thesis 에이전트 자율성을 늘리기보다, 틀릴 수 있는 지점을 프로토콜과 역할 경계로 좁힌다.
Plasma thesis 큰 프롬프트가 아니라 mission-bound MCP read와 source/result 분리로 grounded report를 만든다.
Action thesis Fleet 전체가 아니라 evidence gate, review lane, prompt invariant test만 얇게 차용한다.
control map

정확도 축: Fleet은 목표 고정, 근거 문턱, 결과 검문을 한 흐름으로 묶는다.

토큰 축: Plasma의 문제는 작은 prompt보다 누적 provider session과 tool result transcript에 가깝다.

진단: 두 시스템은 다른 문제를 푼다

Fleet은 여러 에이전트가 같은 저장소에서 일할 때의 정확도를 다룬다. Plasma는 하나의 연구 mission 안에서 근거 있는 답변과 보고서를 만드는 정확도를 다룬다. 그래서 정답은 Fleet을 통째로 옮기는 것이 아니라, 오류 비용이 큰 Plasma 단계에만 작은 검문소를 추가하는 것이다.

Fleet

정확도를 절차로 만든다

Intent Gate, Mode Gate, Mission Anchor, Context Confidence, Result Integrity, Carrier 역할 분리, Wiki 승인 흐름이 한 운영 체계로 묶여 있다.

Plasma

근거 경계를 얇게 유지한다

source는 원자료이고 agent answer/report는 result/artifact다. read-first MCP를 통해 자료를 읽고, 큰 prompt pack이나 report-only corpus를 피한다.

Tradeoff

정확도 장치는 비용을 만든다

Fleet식 gate를 과하게 옮기면 Plasma의 빠른 연구 루프가 무거워진다. 적용 지점은 report, synthesis, session fork처럼 오류와 비용이 커지는 구간이어야 한다.

Fleet의 정확도 스택

아래 항목을 선택하면 Fleet이 어느 지점에서 에이전트 오류를 줄이는지 볼 수 있다. 핵심은 한 번의 멋진 prompt가 아니라, 시작 전·진행 중·결과 후에 반복되는 검문이다.

gate before action

위험도에 맞는 작업 모드를 고른다

모든 operational request는 Intent Gate와 Mode Gate를 지나며, baseline/midline/redline/frontline 중 정확히 하나의 protocol skill을 로드한다. 구조 변경, API 변경, multi-module, doctrine/prompt-policy 작업은 Downward Guard 때문에 낮은 모드로 처리할 수 없다.

  • 작업 난이도와 검증 밀도를 연결한다.
  • 되돌리기 어려운 작업은 risk review와 complete confidence를 요구한다.
  • simple task에는 compact path가 있어 ceremony를 줄인다.

session fork

Plasma가 배울 점: 통째 이식이 아니라 얇은 검문소

Plasma의 장점은 가벼운 C1 loop다. 따라서 Fleet식 장치를 모든 turn에 넣는 대신, report generation, synthesis, long session, source promotion처럼 실패 비용이 큰 지점에만 적용하는 편이 낫다.

Priority 1High

Report 전 evidence gate

보고서 작성 직전에 읽은 source, 핵심 claim, unsupported gap을 짧게 점검한다. Context Confidence의 최소판이다.

Priority 2High

Reviewer session lane

긴 report나 designed HTML export는 writer와 분리된 reviewer가 citation 누락, weak claim 승격, stale source를 검사한다.

Priority 3Medium

Report session fork

리포트 작업은 research session에서 fork해 후속 연구 turn이 report planning/drafting transcript 비용을 계속 지불하지 않게 한다.

Priority 4Medium

Prompt invariant tests

agent output은 source가 아니다, default MCP는 read-first다, mutation tools는 기본 노출하지 않는다는 규칙을 테스트로 잠근다.

최종 판단

Fleet의 교훈은 “강한 프롬프트를 쓰자”가 아니다. 정확도는 목표 고정, 근거 문턱, 역할 경계, 산출물 검증, 장기 기억 승인으로 만들어진다. Plasma는 이 전체 체계를 복제할 필요는 없다. 대신 report 전 evidence gate, 별도 review lane, report session fork, prompt invariant tests를 작은 product control로 추가하는 편이 가장 현실적이다.
