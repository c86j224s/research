한 장 요약

OAuth2

인가 프레임워크입니다. 역할, grant, endpoint 의미, token response, bearer token 사용 같은 상호운용 프로토콜을 정의합니다.

OIDC

OAuth2 위의 identity layer입니다. `openid` scope, ID Token, UserInfo, claims, OP discovery를 정의합니다.

JWT

토큰 포맷입니다. OIDC ID Token은 JWT가 필수이고, OAuth access token은 opaque 또는 JWT 중 선택할 수 있습니다.

JOSE

서명·암호화·키·알고리즘 계층입니다. JWS/JWE/JWK/JWA가 ID Token과 JWT access token 검증의 기반입니다.

의사결정 기준

현대 기본값 Authorization Code + PKCE + TLS + exact redirect URI 검증.

피해야 할 것 Implicit access token front-channel 발급, ROPC, 느슨한 redirect URI matching.

Access Token 선택 opaque + introspection 또는 JWT + RFC 9068 중 운영 모델에 맞춰 선택.

3. 구현 역할별 임무

OAuth2 Authorization Server

필수 축

선택한 grant, authorization endpoint, token endpoint, client authentication, redirect URI 검증, token 발급.

토큰 정책

access token을 opaque로 할지 JWT로 할지 결정. refresh token 발급·회전·재사용 탐지는 정책 영역.

보안 기준

PKCE 지원, exact redirect URI matching, open redirect 금지, TLS, metadata 제공을 기본값으로 둡니다.

OpenID Provider

OIDC Core

`openid` scope를 처리하고, 인증 결과를 ID Token으로 발급합니다.

JWT/JWS

ID Token을 서명하고 JWKS로 공개키를 제공합니다. 필요 시 JWE로 암호화합니다.

Claims

`sub`, `iss`, `aud`, `exp`, `iat`, `nonce`와 `profile`, `email` 같은 표준 claim을 제공합니다.

OAuth Client / OIDC RP

요청 생성

state, nonce, PKCE 값을 생성하고 redirect_uri를 정확히 사용합니다.

ID Token 검증

decode가 아니라 verify입니다. signature, issuer, audience, expiry, nonce를 검증합니다.

세션 생성

ID Token 검증 뒤 자체 앱 세션을 만들고, ID Token을 API 권한 토큰처럼 쓰지 않습니다.

Resource Server

Bearer 처리

`Authorization: Bearer` 헤더와 오류 응답 semantics를 구현합니다.

Opaque

reference token이면 introspection endpoint로 유효성·scope·subject를 확인합니다.

JWT AT

JWT access token이면 RFC 9068 프로파일에 맞춰 iss, aud, exp, signature, typ 등을 검증합니다.

6. 표준이 열어둔 설계 자유

인증 방법

OAuth/OIDC는 “사용자를 어떻게 인증할지”를 강제하지 않습니다. 비밀번호, WebAuthn, MFA, 기업 SSO, risk-based auth는 제품 정책입니다.

토큰 수명

`exp`는 필요하지만 access/refresh token lifetime 값은 배포 환경의 위험도와 UX 요구에 따라 결정됩니다.

Endpoint path

`/authorize`, `/token` 같은 path는 고정 표준이 아닙니다. discovery metadata의 endpoint 값을 따릅니다.

Scope 의미

`openid`, `profile`, `email` 같은 OIDC 표준 scope 외 업무 scope의 의미는 서비스가 정합니다.

Consent와 약관

동의 개념은 흐름에 등장하지만 화면 구성, 문구, 재동의 조건, 감사 로그는 구현체 정책입니다.

권한 판단

Resource Server는 token 검증 후에도 tenant, role, ownership, business rule을 적용해야 합니다. 이 부분은 OAuth token profile만으로 완성되지 않습니다.

7. 용어 검색

OAuth2

제3자 client가 제한된 API 접근권을 얻는 인가 프레임워크.

OIDC

OAuth2 위에서 사용자 인증 결과와 identity claims를 표준화한 layer.

JWT

claims를 compact JSON 기반 토큰으로 표현하는 포맷.

JWS

payload의 무결성과 발급자 진위를 검증하기 위한 서명/MAC 컨테이너.

JWKS

공개키 집합을 JSON으로 제공해 client/API가 JWT 서명을 검증하게 하는 endpoint.

PKCE

authorization code 탈취만으로 token 교환을 못 하게 하는 proof key 메커니즘.

nonce

OIDC authentication request와 ID Token을 묶어 replay/token substitution을 줄이는 값.

state

redirect 기반 흐름에서 CSRF와 요청-응답 상관관계를 보호하는 값.

Introspection

opaque access token의 active 여부와 metadata를 Authorization Server에 확인하는 방식.

8. 근거와 추적성

주장근거 문서구현 의미
OAuth2는 인가 프레임워크다.RFC 6749역할, grant, endpoint, token 발급 흐름을 구현한다.
Bearer token 사용법은 별도 문서다.RFC 6750API는 Authorization header와 오류 응답을 표준 방식으로 처리한다.
PKCE는 현대 redirect flow의 기본 방어선이다.RFC 7636, RFC 9700Authorization Server는 PKCE를 지원하고 token endpoint에서 검증한다.
OAuth endpoint path는 고정이 아니라 metadata로 발견한다.RFC 8414`authorization_endpoint`, `token_endpoint`, `jwks_uri` 값을 사용한다.
OIDC는 OAuth2 위의 identity layer다.OpenID Connect Core 1.0`openid` scope, ID Token, UserInfo, claims를 구현한다.
ID Token은 JWT다.OIDC Core ID Token, RFC 7519JSON payload decode가 아니라 JWS/JWE 검증 후 신뢰한다.
JWT access token은 선택 프로파일이다.RFC 9068opaque token을 쓰면 필수가 아니며, JWT AT 선택 시 적용한다.
OIDC discovery와 registration은 OpenID Foundation 사양이다.OIDC Discovery 1.0, OIDC Dynamic Client Registration 1.0OP metadata 제공과 dynamic client registration 범위를 선택한다.

이 문서는 외부 스크립트, 외부 스타일시트, 이미지 의존성, 외부 앵커 없이 동작합니다. 근거는 표준 문서명과 섹션 단위로 추적합니다.
