🗺️ OAuth 2.0과 OIDC의 관계

OAuth 2.0 — 위임 권한 부여

"이 앱이 내 구글 드라이브에 접근해도 됩니다"

  • 리소스 접근 권한(Authorization) 위임 프레임워크
  • Access Token 발급이 핵심
  • "누구인가"를 직접 정의하지 않음
  • RFC 6749, 6750, 7636 등

OIDC — 신원 확인 (인증 레이어)

"이 사람이 kim@example.com입니다"

  • OAuth 2.0 위에 신원(Authentication) 레이어 추가
  • ID Token (JWT) 발급이 핵심
  • 사용자 정보를 표준화된 Claims로 전달
  • OpenID Connect Core 1.0 스펙
💡
핵심 관계
OIDC는 OAuth 2.0을 대체하지 않고 확장합니다. OIDC를 구현하면 OAuth 2.0도 자동으로 구현됩니다. Authorization Server가 OIDC를 지원하면 OpenID Provider (OP)라고 부릅니다.

⚠️
HTTP + 앞단 프록시 구성 시 주의
AS 자체는 HTTP로 동작하지만, OAuth 스펙은 HTTPS를 강제합니다. 앞단 LB/Proxy가 TLS를 처리하므로 AS는 X-Forwarded-Proto: https 헤더를 신뢰하고, Redirect URI 검증 시 scheme을 https로 강제해야 합니다. 또한 Secure 쿠키 플래그는 앞단이 처리하거나, 내부적으로 플래그를 설정해도 됩니다 (프록시가 전달).

Implicit Grant & ROPC — 사용 금지
OAuth 2.1 (draft)에서 공식 제거. Implicit은 Access Token이 URL fragment에 노출되고, ROPC는 비밀번호를 Client에 직접 전달하여 보안상 심각한 문제가 있습니다. 새 구현에서는 절대 구현하지 마세요.

PKCE 핵심 검증 로직
// Client 측: code_verifier 생성
code_verifier = base64url(random_bytes(32))  // 43 chars after encoding

// Client 측: code_challenge 생성
code_challenge = base64url(sha256(code_verifier))

// AS 측: Token Endpoint에서 검증
expected_challenge = base64url(sha256(received_code_verifier))
if expected_challenge != stored_code_challenge {
    return error("invalid_grant")
}

Client Credentials에는 refresh_token이 없습니다
만료되면 Client가 직접 새 Access Token을 요청합니다. Client Secret이 곧 자격증명이므로 Refresh Token이 불필요합니다.

⚠️
Refresh Token Rotation 권장
Refresh Token을 사용할 때마다 새 Refresh Token을 발급하고, 이전 것은 폐기하는 Rotation 전략을 사용하면 탈취된 토큰을 감지할 수 있습니다 (이전 토큰으로 재사용 시 알람).

🔄 Access Token 형식: JWT vs Opaque
JWT Access Token (자가포함)
  • 장점: RS가 AS에 질의 없이 검증 가능 (오프라인 검증)
  • 장점: 확장성 좋음, 분산 시스템에 적합
  • 단점: 즉시 폐기 불가 (exp까지 유효)
  • 단점: Payload에 민감 정보 노출 주의
  • 대안: 짧은 exp + Refresh Token 조합
  • RFC 9068에서 표준 Claims 정의
Opaque Access Token (불투명)
  • 장점: 즉시 폐기 가능 (DB에서 삭제)
  • 장점: 토큰 자체에 정보 없음 — 탈취 시 단독으로 의미 없음
  • 단점: RS가 매 요청마다 AS에 Introspection 필요
  • 단점: AS가 단일 장애점 될 수 있음
  • Redis 같은 빠른 저장소에 저장
✅
우리 구현 권장: JWT + 짧은 만료 + Introspection 선택 지원
JWT Access Token으로 RS 부하 줄이고, 긴급 폐기가 필요하면 jti 블랙리스트 또는 Introspection 엔드포인트로 처리. Refresh Token은 Opaque(DB 저장)로 구현.

🪪 ID Token — OIDC의 핵심
ℹ️
ID Token은 Access Token이 아닙니다
ID Token은 사용자가 누구인지 Client에게 알려주는 용도입니다. API 호출에 ID Token을 쓰면 안 됩니다 (Access Token을 써야 합니다). ID Token은 Client가 검증 후 내부적으로 세션을 만드는 데 사용합니다.

✅ ID Token 검증 체크리스트 (Client/RP 측)
  1. alg 검증 — none 절대 거부. 예상 알고리즘과 일치 확인.
  2. 서명 검증 — JWKS 엔드포인트에서 kid 기반으로 공개키 조회 후 서명 확인.
  3. iss 검증 — Discovery 문서의 issuer와 정확히 일치해야 함.
  4. aud 검증 — 내 client_id가 포함되어 있어야 함.
  5. exp 검증 — 현재 시간 이후여야 함 (약간의 클럭 스큐 허용 가능, 보통 5분).
  6. iat 검증 — 너무 오래된 토큰 거부 (선택적).
  7. nonce 검증 — 보냈다면 받은 값과 일치해야 함 (리플레이 방지).
  8. at_hash 검증 — Access Token이 함께 왔다면 검증.
  9. azp 검증 — aud가 여러 개면 azp가 내 client_id인지 확인.

🛡️ 절대 지켜야 할 보안 요건
1. PKCE — 모든 Authorization Code 요청에 필수 ▶

Public Client는 물론 Confidential Client에도 PKCE를 강제하는 것이 OAuth 2.1의 방향입니다. code_challenge_method는 반드시 S256만 허용해야 합니다. plain은 중간자 공격에 무의미합니다. 

// AS에서 plain 거부
if req.CodeChallengeMethod == "plain" {
    return ErrInvalidRequest("only S256 is supported")
}
2. Redirect URI — 정확한 일치 검증 (prefix 매칭 금지) ▶

등록된 redirect_uri와 요청의 redirect_uri는 완전 일치(exact match)해야 합니다. prefix 매칭, 와일드카드, 정규식은 Open Redirect 공격으로 이어집니다.

🚫
금지 패턴
https://app.example.com을 등록했을 때 https://app.example.com.evil.com을 허용하면 안 됩니다. scheme, host, port, path 모두 정확히 일치해야 합니다.
3. state — CSRF 방지 (필수 구현) ▶

Client가 생성한 충분히 랜덤한 state를 세션에 저장하고, callback에서 반환된 state와 정확히 비교해야 합니다. AS는 state 파라미터를 변경 없이 그대로 redirect_uri에 전달해야 합니다.

4. Authorization Code — 단 한 번만 사용 가능 ▶

Authorization Code는 일회용입니다. 재사용 시도가 감지되면:

  1. 즉시 에러 반환
  2. 해당 code로 이미 발급된 Access Token/Refresh Token을 모두 폐기 (공격 감지 대응)
  3. 보안 이벤트 로그 기록

Authorization Code의 유효 기간: 10분 이하 권장 (실제로는 1~2분).

5. JWT alg=none 거부 — 서명 검증 강제 ▶

JWT를 파싱할 때 항상 예상 알고리즘을 명시적으로 지정해야 합니다. 라이브러리에 자동 알고리즘 감지를 허용하면 alg:none 공격에 취약해집니다. 

// golang-jwt/jwt 예시
token, err := jwt.Parse(tokenStr, keyFunc,
    jwt.WithValidMethods([]string{"RS256", "ES256"}),  // 명시적 지정
)
6. Token Endpoint — 응답에 Cache-Control: no-store 필수 ▶

Token 응답은 절대 캐시되면 안 됩니다. RFC 6749 §5.1에서 명시적으로 요구합니다. 

w.Header().Set("Cache-Control", "no-store")
w.Header().Set("Pragma", "no-cache")
7. HTTPS 강제 (앞단 프록시 환경에서) ▶

내부 AS는 HTTP지만, 외부로는 반드시 HTTPS여야 합니다. 구현 시 주의사항:

  • Redirect URI 등록/검증 시 https:// scheme 강제
  • Discovery 문서의 issuer, 엔드포인트 URL은 https://
  • X-Forwarded-Proto 헤더로 실제 scheme 판별 (신뢰할 프록시만)
  • localhost 예외: 개발용으로만, 프로덕션에선 절대 금지
8. Refresh Token — 안전한 보관 및 Rotation ▶
  • Refresh Token은 DB에 저장 (JWT 아님). Hash 형태로 저장 권장.
  • 사용 시마다 새 Refresh Token 발급 + 이전 것 즉시 폐기 (Rotation)
  • 이전 Refresh Token으로 요청 시 → 해당 세션 전체 폐기 (탈취 감지)
  • Binding: user_id, client_id, scope를 함께 저장. 다른 client로 사용 불가.
9. Rate Limiting — 브루트포스 방지 ▶
  • 로그인 엔드포인트: IP + 계정별 실패 횟수 제한
  • Token Endpoint: client_id별 요청 수 제한
  • TOTP: 연속 실패 시 계정 잠금 (짧은 유예 시간 포함)
  • 표준에 규정 없음 — 구현 자율이지만 반드시 필요
10. scope 검증 — 요청 스코프가 허용 범위 내인지 ▶
  • Client가 요청한 scope는 그 Client에 허용된 scope 내여야 함
  • Refresh Token으로 갱신 시 원래 scope보다 넓은 scope 요청 불가
  • 발급된 scope가 요청과 다르면 응답에 실제 scope 포함 필수
🎨 표준이 자율에 맡기는 부분

RFC가 결과(무엇을)는 정의하지만, 방법(어떻게)은 구현자에게 맡기는 영역들입니다.

Access Token 포맷

JWT(자가포함) 또는 Opaque(불투명 문자열) 모두 허용. RS가 어떻게 검증할지는 AS가 결정.

Token 저장 방식

PostgreSQL, Redis, 메모리 어느 것이든 가능. Refresh Token을 DB에, Access Token(JWT)을 stateless로 하는 하이브리드도 일반적.

Authorization Code 생성 방식

포맷은 자유. crypto/rand로 생성한 충분한 길이의 랜덤 문자열이면 됨 (최소 128비트 엔트로피).

세션 관리

로그인 세션을 쿠키로 유지할지, 매번 재인증할지, 세션 시간을 얼마나 할지 표준에 없음.

Consent 화면 UX

어떻게 보여줄지, 어떤 정보를 표시할지, 기억 여부(rememberConsent) 등은 구현자 재량.

scope 정의 및 계층 구조

표준은 openid, profile, email, address, phone, offline_access 정도만 정의. 나머지는 자유롭게 정의 가능. (e.g. read:users, admin:all, api:write)

Custom Claims

JWT Payload에 표준 Claims 외 커스텀 데이터 추가 가능. 충돌 방지를 위해 네임스페이스 URL 형태 권장: "https://api.example.com/roles": ["admin"]

Token 수명 정책

Access Token exp, Refresh Token exp, Authorization Code exp를 얼마로 할지 구현자 결정. 단, Authorization Code는 최대 10분 이하 권장.

동적 클라이언트 등록 (RFC 7591)

API로 Client를 동적 등록하는 기능. 지원 여부는 선택. 어드민 UI로만 등록하는 것도 완전히 합법.

Introspection 응답 필드

RFC 7662의 active 필드는 필수, 나머지(scope, username, exp 등)는 선택적으로 포함.

🧩 표준에 없어서 우리가 직접 설계해야 하는 것들
계정 관리 (User Management)

OAuth/OIDC는 "로그인 후 토큰 발급"만 다루고, 계정 생성/수정/삭제는 전혀 다루지 않습니다.

  • 회원가입, 이메일 인증 플로우
  • 비밀번호 정책 (길이, 복잡도, 재사용 금지)
  • 비밀번호 분실/재설정 (이메일 링크 / 임시 코드)
  • 계정 잠금 정책 (실패 횟수, 잠금 해제)
  • 이메일 변경 플로우
  • 계정 탈퇴 및 데이터 삭제
TOTP / MFA 관리

RFC 6238(TOTP), RFC 4226(HOTP)은 알고리즘만 정의. 통합 방법은 표준 없음.

  • TOTP 등록 (QR코드 생성, otpauth:// URI 형식)
  • 복구 코드 (Backup Codes) 생성 및 관리
  • TOTP 해제 및 재설정 플로우
  • Time skew 허용 (±1 step = ±30초)
  • TOTP 코드 재사용 방지 (마지막 성공 코드 기록)
  • amr claim에 otp 포함하여 AS가 MFA 완료 표시
  • acr을 통해 RP가 MFA 요구 가능 (e.g. acr_values=mfa)
// otpauth URI 형식 (QR코드로 인코딩)
otpauth://totp/{issuer}:{account}?secret={base32_secret}&issuer={issuer}&algorithm=SHA1&digits=6&period=30
Admin UI / API
  • Client 등록/수정/삭제/조회
  • Client의 허용 Grant Type, Redirect URI, Scope 관리
  • Client Secret 순환 (rotate)
  • 사용자 목록, 상태, 역할 관리
  • 발급된 토큰/세션 목록 및 강제 폐기
  • 스코프 정의 및 설명 관리
  • 감사 로그 조회
  • JWKS 키 순환 관리
RBAC / 권한 체계

OAuth scope는 접근 범위이지 역할(Role) 시스템이 아닙니다. 둘을 구분해 설계해야 합니다.

  • 사용자 역할(Role): admin, user, moderator 등
  • OAuth scope: read:profile, write:posts 등 API 수준 접근 제어
  • Client별 허용 scope 화이트리스트
  • 역할 정보를 JWT Custom Claim으로 포함 가능
  • RS는 scope + role 양쪽으로 권한 검증 가능
감사 로그 (Audit Log)

표준에 전혀 없지만, 보안 및 컴플라이언스를 위해 필수입니다.

  • 기록 대상: 로그인 시도(성공/실패), 토큰 발급, 토큰 폐기, Consent, 관리자 작업
  • 포함 정보: 타임스탬프, 사용자 ID, Client ID, IP, User-Agent, 결과
  • 로그 불변성 보장 (삭제/수정 불가)
  • 보존 기간 정책
키 순환 (Key Rotation)

JWT 서명 키를 주기적으로 교체하는 전략. JWKS에서 kid로 관리.

  • 새 키 추가 → 이전 키로 발급된 토큰이 만료될 때까지 이전 키도 JWKS에 유지
  • JWKS는 여러 키를 담을 수 있음 — 전환 기간 동안 병행 운영
  • 캐시 무효화: RS의 JWKS 캐시를 언제 갱신할지 전략 필요
  • kid 기반 조회: AS가 kid를 JWT Header에 포함 → RS가 맞는 키로 검증
비밀번호 저장

표준 외 영역이지만 구현 방법은 명확합니다.

  • 반드시 단방향 해시 + 솔트: bcrypt(cost≥12), argon2id(권장), scrypt
  • MD5, SHA-1, SHA-256 평문 해시 금지
  • Golang: golang.org/x/crypto/bcrypt 또는 golang.org/x/crypto/argon2
로그인 페이지 — HTML vs API

AS가 직접 HTML 로그인 페이지를 서빙할지, 별도 프론트엔드가 API로 처리할지는 구현 선택사항입니다.

  • AS 직접 서빙: 단순하고 쿠키 기반 세션 관리 용이. 많은 AS가 이 방식 (Keycloak, Hydra+Kratos 등)
  • 분리형: AS는 API만, 별도 로그인 프론트엔드가 AS API 호출. UX 커스터마이징 자유도 높음.

핵심 데이터 모델
// Client
type Client struct {
    ID                  string
    Secret              string    // bcrypt hash. Public Client면 빈 값.
    Type                string    // "confidential" | "public"
    RedirectURIs        []string  // 허용된 redirect URI 목록 (exact match)
    AllowedGrantTypes   []string
    AllowedScopes       []string
    RequirePKCE         bool
    RequireConsent      bool
    CreatedAt           time.Time
}

// AuthorizationCode (Redis에 저장, TTL 2분)
type AuthorizationCode struct {
    Code                string
    ClientID            string
    UserID              string
    RedirectURI         string
    Scope               string
    CodeChallenge       string
    CodeChallengeMethod string
    Nonce               string
    ExpiresAt           time.Time
    Used                bool      // 재사용 방지
}

// RefreshToken (PostgreSQL에 저장)
type RefreshToken struct {
    TokenHash   string    // SHA-256 해시 저장 (평문 저장 금지)
    ClientID    string
    UserID      string
    Scope       string
    ExpiresAt   time.Time
    CreatedAt   time.Time
    RevokedAt   *time.Time
    FamilyID    string    // Rotation 감지용 (같은 lineage)
}
JWT Access Token 발급 예시
func (s *TokenService) IssueAccessToken(userID, clientID, scope string) (string, error) {
    now := time.Now()
    claims := jwt.MapClaims{
        "iss":       s.config.Issuer,
        "sub":       userID,
        "aud":       s.config.Issuer + "/api",  // 또는 Resource Server URL
        "exp":       now.Add(s.config.AccessTokenTTL).Unix(),
        "iat":       now.Unix(),
        "jti":       uuid.New().String(),
        "client_id": clientID,
        "scope":     scope,
    }

    token := jwt.NewWithClaims(jwt.SigningMethodRS256, claims)
    token.Header["kid"] = s.keys.CurrentKeyID()

    return token.SignedString(s.keys.CurrentPrivateKey())
}
TOTP 검증 예시
import "github.com/pquerna/otp/totp"

func (s *AuthService) VerifyTOTP(userSecret, code string) error {
    valid := totp.Validate(code, userSecret)
    if !valid {
        return ErrInvalidTOTP
    }
    // 재사용 방지: 마지막 성공 코드를 Redis에 저장 (TTL 30초)
    if s.cache.IsTOTPCodeUsed(userSecret, code) {
        return ErrTOTPReplay
    }
    s.cache.MarkTOTPCodeUsed(userSecret, code)
    return nil
}

  5. TOTP: 등록, 검증, 복구 코드
  6. Authorization Code + PKCE: /authorize, Consent, Code 발급
  7. Token Endpoint: code→token 교환, Refresh Token
  8. OIDC: ID Token 발급, /userinfo, Discovery
  9. Client Credentials Grant
  10. Introspection + Revocation
  11. Admin API/UI
  12. 감사 로그, 모니터링
