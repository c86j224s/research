OAuth2 · OIDC · JWT 구현 전술도

OAuth2.0, OpenID Connect, JWT/JOSE를 “프로토콜이 정한 것”과 “제품이 결정할 것”으로 분리한 재작성판입니다. 기존 HTML은 보존하고 이 문서는 v2 파일로 생성되었습니다.

핵심 판단

OAuth2 인가 프레임워크입니다. 로그인 그 자체보다 API 접근권 위임을 다룹니다.

OIDC OAuth2 위에 사용자 신원 확인을 얹은 identity layer입니다.

JWT claim을 담는 토큰 포맷입니다. ID Token은 JWT, access token은 선택입니다.

JOSE JWT의 서명, 암호화, 키, 알고리즘 표준군입니다.

현대 기본값

  • Authorization Code Flow + PKCE S256
  • TLS와 exact redirect URI matching
  • `state`와 OIDC `nonce` 검증
  • metadata discovery 사용: endpoint path 추측 금지
  • ID Token은 decode가 아니라 verify 후 신뢰: 허용 `alg`, issuer JWKS, `iss`, `aud`, `exp`, `nonce`까지 확인

Implicit / front-channel token 발급

OIDC Core에는 존재하지만, 현대 보안 기준에서는 access token을 authorization response로 직접 전달하는 패턴을 피하는 것이 기본입니다. 새 구현은 Code + PKCE를 우선하십시오.

핵심 위험: URL fragment/브라우저 경로 노출, token leakage, replay 대응 한계.

Hybrid Flow

`code id_token`, `code token`, `code id_token token`처럼 일부 결과를 authorization endpoint에서 받고, code를 token endpoint에서 교환합니다. 특수 profile 호환 목적이 아니면 Code + PKCE가 더 단순하고 안전한 기본값입니다.

Hybrid를 구현한다면 ID Token의 signature/claims/nonce뿐 아니라 front-channel code와 token binding을 위해 `c_hash`, `at_hash` 검증을 포함하십시오.

3. 구현 역할

Core

Authorization Server

grant, endpoint, token response, client auth, redirect URI 검증을 담당합니다.

Token

Token 정책

access token 형식, refresh token 발급·회전·재사용 탐지는 배포 정책입니다.

Security

방어선

PKCE, exact matching, open redirect 차단, TLS, metadata 제공을 기본값으로 둡니다.

OIDC

OpenID Provider

`openid` 요청을 처리하고 사용자를 인증한 결과를 ID Token으로 발급합니다.

JOSE

서명과 JWKS

ID Token을 JWS로 서명하고 공개키를 JWKS로 제공합니다. 필요 시 JWE를 씁니다.

Claims

Claims 정책

`sub`, `iss`, `aud`, `exp`, `iat`, `nonce`, profile/email claim을 제공합니다.

Request

요청 준비

state, nonce, PKCE 값을 생성하고 redirect_uri를 등록값과 맞춥니다.

Verify

ID Token 검증

payload JSON decode만으로 신뢰하지 않고 signature와 claims를 검증합니다.

Session

앱 세션

검증 성공 후 자체 세션을 만들고 ID Token을 API 권한으로 쓰지 않습니다.

Bearer

Token 수신

`Authorization: Bearer` 처리와 오류 semantics를 구현합니다.

Opaque

Introspection

opaque token이면 introspection으로 active/scope/subject를 확인합니다.

JWT AT

Local verify

JWT access token이면 RFC 9068 기반으로 allowed alg, issuer-bound JWKS, signature, iss, aud, exp, typ 등을 검증합니다.

5. Access Token 전략 선택

Opaque / Reference Token

  • Resource Server가 introspection으로 검증
  • 중앙 폐기와 상태 관리가 쉬움
  • 온라인 검증 또는 캐시 전략 필요

JWT Access Token

  • Resource Server가 서명과 claims를 로컬 검증
  • 분산 검증에 유리
  • audience, key rotation, expiry, claim 설계가 중요

6. 표준이 비워둔 영역

인증 방법

비밀번호, 패스키, MFA, 기업 SSO, risk-based auth는 제품 정책입니다.

Scope 의미

표준 OIDC scope 외 업무 scope의 의미는 서비스가 정합니다.

권한 판단

token 검증 후에도 tenant, role, ownership, business rule을 적용해야 합니다.

Token lifetime

만료 시간 값과 refresh token 정책은 위험도와 UX에 맞춰 결정합니다.

Endpoint path

`/authorize`, `/token` path는 고정 표준이 아니며 metadata 값을 사용합니다.

Consent UX

동의 화면, 문구, 재동의 조건, 감사 로그는 구현체 정책입니다.
